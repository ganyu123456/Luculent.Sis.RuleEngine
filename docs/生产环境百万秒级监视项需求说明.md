| 指标 | 目标值 |
|------|:---:|
| 监视项容量 | 1,000,000 |
| 检测周期 | 1s ~ 60s（可配置，支持混部） |

## 三、前置规则：MonitorCenter vs Rule Engine 对比

### 3.1 MonitorCenter 前置规则架构

MonitorCenter 的前置规则是一个**独立的、带自己规则引擎的子系统**。

```
MonitorItemPipeline
  ├─ PreruleCheckBlock
  │    ├─ 读 monitor.PreruleId
  │    ├─ 查 PreruleStore.GetPreruleState(preruleId)
  │    │    ├─ state == true  → 满足前置条件，继续计算
  │    │    ├─ state == false → 不满足，跳过计算，写入"不满足前置条件"事件
  │    │    └─ state == null  → 首次评估前，跳过本次周期
  │    └─ 并行运行:

  PrerulePipeline（独立管道，主动评估前置规则）
  ├─ RuleCalculateBlock → 读数据源 + 执行规则计算
  └─ PreruleCacheBlock → 缓存结果到 PreruleStore (true/false)
```

**PreruleStore 核心设计**：

| 设计项 | 实现 |
|------|------|
| 数据结构 | 64 分片 `ConcurrentDictionary<string, bool?>` |
| 分片方式 | `hash(key) & 63` 位运算，无锁竞争 |
| 状态语义 | `true` = 满足条件，`false` = 不满足，`null` = 尚未评估 |
| 初始化 | 禁用规则 → 直接置 `true`（视为满足）；启用规则 → 等待 PrerulePipeline 首次评估 |

**PreruleCacheItem 结构**（对标监视项配置，但用于前置规则评估）：

```
PreruleCacheItem:
  ├─ Id, Name, Desc
  ├─ RefreshIntervalSecond  ← 独立刷新周期
  ├─ RuleType               ← 前置规则本身有规则类型
  ├─ IsEnabled
  ├─ FocusSourceId
  ├─ MonitorSources[]       ← 前置规则自己的数据源
  └─ 关联规则子实体:
       ├─ RuleExpression (Expression)
       ├─ RuleRangeDurations (RangeDuration)
       ├─ RuleRangeFrequencies (RangeFrequency)
       └─ RulePackageValues (RulePackageValue)
```

### 3.2 Rule Engine 当前前置规则实现

Rule Engine 只实现了 **3 种硬编码检查**，没有独立的前置规则配置和评估系统。

```
WorkerCalculationService.ProcessMonitorAsync
  ├─ ① PrerulePipeline.CheckAsync(monitor)
  │    ├─ ManualFlag 检查          (monitor.ManualFlag == 0 → suppress)
  │    ├─ StopMonitor 检查         (Redis 查 StopMonitorKey 是否在报警)
  │    └─ SourceDependency 检查    (Redis 查每个 SourceType=1 的 RelatedId)
  │
  ├─ ② RuleDispatcher.CalculateAsync  (规则计算)
  └─ ③ 报警写入
```

**PreruleConfig（当前）**：
```csharp
public class PreruleConfig
{
    public bool IsEnabled { get; set; } = true;
    public bool EnableManualFlagCheck { get; set; } = true;
    public bool EnableStopMonitorCheck { get; set; } = true;
    public bool EnableSourceDependencyCheck { get; set; }
}
```

### 3.3 差距分析

| 对比项 | MonitorCenter | Rule Engine | 差距 |
|------|:---:|:---:|:---:|
| **前置规则类型** | Expression / RangeDuration / RangeFrequency / FeatureValue / PackageValue / InterfaceMonitoring | 只有 3 种硬编码检查 | MonitorCenter 的前置规则是有独立 RuleType 的可配置规则，Rule Engine 缺失 |
| **前置规则数据源** | 前置规则**有自己的数据源**，独立于监视项的数据源 | 无 | 缺失——ManualFlag 不需要数据源，但 SourceDependency 只能查 Redis 报警状态 |
| **前置规则评估管道** | PrerulePipeline (TPL Dataflow): RuleCalculate → PreruleCache | 无——3 个检查在 ProcessMonitorAsync 入口串行执行 | 缺失 |
| **前置规则状态缓存** | PreruleStore: 64 分片 ConcurrentDictionary，独立于监视项状态 | 无——ManualFlag 直接读 monitor 属性，StopMonitor/SourceDependency 依赖 Redis 报警状态 | 缺失 |
| **刷新间隔** | 前置规则有独立 RefreshIntervalSecond | 无——每周期都检查 | 非瓶颈，纯内存操作 |
| **监控项关联** | 监控项通过 `PreruleId` 引用前置规则，多个监视项可共享同一前置规则 | 无——3 个检查是 MonitorConfig 的内联属性 | MonitorCenter 支持复用，Rule Engine 每个监控项独立配置 |
| **ManualFlag** | 通过 `monitor.ManualFlag` 字段，值 0=停止, 1=运行 | 相同实现 | 一致 |
| **StopMonitor** | 通过 `monitor.StopMonitorKey`，查询关联监视项是否在报警状态 | 相同实现（查 Redis `GetAlarmAsync`） | 一致，但 MonitorCenter 查本地内存 Store |
| **SourceDependency** | 通过 `MonitorSourceDefinition.SourceType == 1` 的 `RelatedId` 查询关联监视项报警 | 相同实现（查 Redis `GetAlarmAsync`） | 逻辑一致 |

### 3.4 结算

**三层前置规则（当前 Rule Engine 已实现）不是 MonitorCenter 前置规则的全部**。MonitorCenter 有一个完整的前置规则评估子系统——前置规则本身就是一个独立的规则引擎实例，有自己的 RuleType、DataSources、RuleExpressions 等。

但在百万秒级场景下，是否需要立即实现完整前置规则评估系统？

| 三种硬编码检查 | MonitorCenter 完整前置规则 |
|:---:|:---:|
| 满足 ManualFlag/StopMonitor/SourceDependency 场景 | 支持 Expression/RangeDuration 等复杂前置条件 |
| 零额外状态存储 | 需要 PreruleStore + PrerulePipeline |
| 纯内存操作，无计算开销 | 前置规则评估 = 一次完整规则计算 |
| 配置从 MonitorItem 字段直接映射 | 需要独立的 Prerule 实体同步 |

**建议**：三层硬编码检查覆盖了生产 95% 的前置规则场景（启停控制、关联抑制、数据源依赖）。完整前置规则评估系统的 Migration 放到第二期，等事件流模型稳定后再做。

## 四、核心改造：事件流模型

### 4.1 当前模型

```
触发 → INSERT trigger 行 (clear_time = NULL)
消除 → INSERT clear 行 + ALTER TABLE UPDATE trigger 行 (clear_time = 消除时间)
                                ↑
                    ClickHouse 实时 mutation，单次 100-500ms
                    大量并发时 mutation 队列爆炸
```

### 4.2 目标模型：状态变更流

```
每次状态变化 INSERT 一行:
  { monitor_id, status_key, occur_time, trigger_value, unit, job_id }

规则: 当前周期状态 != 上一周期状态 → INSERT 一行
      当前周期状态 == 上一周期状态 → 不写入

无 trigger/clear 区分，无 ALTER TABLE UPDATE，纯追加写
```

### 4.3 Worker 写入逻辑（改造后）

```csharp
var result = await _dispatcher.CalculateAsync(monitor, values, now);
var state = await _stateStore.GetAsync(monitor.Id);
var newStatus = result.HasEvent ? result.State : "";

if (newStatus != state.PreviousStatus)
{
    await _alarmWriter.WriteEventAsync(new AlarmEvent
    {
        MonitorId = monitor.Id,
        MonitorKey = monitor.Key,
        StatusKey = newStatus,
        OccurTime = now,
        TriggerValue = result.TriggerValue ?? 0,
        Unit = unit,
        JobId = $"{WorkerId}_{monitor.Key}_{now:yyyyMMddHHmmss}",
    });

    state.PreviousStatus = newStatus;
    await _stateStore.SaveAsync(monitor.Id, state);
}

// 实时报警更新
if (newStatus != "")
    await _alarmWriter.WriteRealtimeAlarmAsync(alarm);
else
    await _alarmWriter.ClearRealtimeAlarmAsync(monitor.Id);
```

### 4.4 一致性收益

| 场景 | 旧模型 (trigger/clear) | 新模型 (状态变更流) |
|------|:---:|:---:|
| StateStore 丢失 | clear 不再生成 → trigger 孤悬 | 下一周期重写相同状态（幂等） |
| Worker 漂移 | trigger 已写但无 clear → 孤悬 | 新 Worker 从 ClickHouse 查最后事件恢复状态 |
| 毫秒级状态抖动 | trigger/clear 配对错乱 | 每条独立，查询侧 LEAD 自然配对 |
| Channel 丢事件 | 丢一个配对断裂 | 最多丢一条中间事件，下一个自动补上 |

### 4.5 查询侧配对（LEAD 窗口函数）

```sql
SELECT
    monitor_id, monitor_key,
    status_key,
    occur_time AS start_time,
    LEAD(occur_time) OVER (
        PARTITION BY monitor_id ORDER BY occur_time
    ) AS end_time,
    trigger_value
FROM alarm_events
WHERE occur_time BETWEEN {startTime} AND {endTime}
  AND monitor_key IN ({filter})
  AND status_key != ''
ORDER BY occur_time DESC
LIMIT 100 OFFSET 0
```

## 五、Worker 性能优化

### 5.1 Task.WhenAll → Parallel.ForEachAsync

```csharp
// 旧: N 个 Task 堆分配
var tasks = dueMonitors.Select(m => ProcessMonitorAsync(m, values, now, ct));
await Task.WhenAll(tasks);

// 新: 分块并行，Task 数 = CPU 核数
await Parallel.ForEachAsync(dueMonitors,
    new ParallelOptions { MaxDegreeOfParallelism = Environment.ProcessorCount },
    (m, ct) => ProcessMonitorAsync(m, values, now, ct));
```

| 写法 | Task 数 | 多核 | GC 压力 |
|------|:---:|:---:|:---:|
| Task.WhenAll | N | 是 | 高 |
| **Parallel.ForEachAsync** | **核数** | **是** | **低** |

### 5.2 批量状态读写

```csharp
// 旧: N 次独立 GetAsync/SaveAsync
var state = await _stateStore.GetAsync(monitor.Id);
await _stateStore.SaveAsync(monitor.Id, state);

// 新: 周期开始时批量加，结束时批量存
var states = await _stateStore.GetBatchAsync(dueMonitorIds);
// ... for 循环计算 ...
await _stateStore.SaveBatchAsync(states);
```

### 5.3 单 Worker 容量推算

| 监视项/Worker | 刷新间隔 | 到期/周期 | 单周期耗时 | 可行性 |
|---:|:---:|:---:|:---:|:---:|
| 50,000 | 30s | ~1,667 | < 1ms | 轻松 |
| 50,000 | 1s | 50,000 | 10-20ms | Parallel 多核 |

## 六、ClickHouse 表设计（事件流模型）

```sql
CREATE TABLE ruleengine.alarm_events
(
    monitor_id      String,
    monitor_key     String,
    monitor_name    String,
    status_key      String,
    status_name     Nullable(String),
    occur_time      DateTime64(3, 'UTC'),
    trigger_value   Float64,
    threshold_value Nullable(Float64),
    rule_type       UInt8,
    config_version  DateTime64(3, 'UTC'),
    worker_id       String,
    shard_id        UInt8,
    unit            Nullable(String),
    job_id          Nullable(String),
    date            Date MATERIALIZED toDate(occur_time)
)
ENGINE = MergeTree()
PARTITION BY toYYYYMMDD(date)
ORDER BY (monitor_id, status_key, occur_time)
TTL date + INTERVAL 90 DAY
SETTINGS index_granularity = 8192;
```

| 设计项 | 选择 | 原因 |
|------|------|------|
| ORDER BY | `(monitor_id, status_key, occur_time)` | `PARTITION BY monitor_id` 窗口函数走自然排序 |
| 无 event_type | — | 状态变更流不区分 trigger/clear |
| 无 clear_time | — | 区间由 LEAD 查询侧配对生成 |

## 七、Worker 启动与恢复

```
Worker 启动
  ├─ ① 注册到 Master
  ├─ ② 从本地 StateStore 加载
  └─ ③ 本地缺失 → ClickHouse 批量查最后事件:
       SELECT monitor_id, status_key
       FROM alarm_events
       WHERE monitor_id IN ({ids})
         AND ROW_NUMBER() OVER (PARTITION BY monitor_id ORDER BY occur_time DESC) = 1
```
