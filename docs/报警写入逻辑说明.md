# 报警写入逻辑说明

## 一、架构总览

```
WorkerCalculationService (计算循环)
  └─ IAlarmWriter (报警写入接口)
       ├─ ProductionAlarmWriter (生产环境，复合)
       │    ├─ RedisAlarmWriter     ← 实时报警 (realtime)
       │    └─ ClickHouseAlarmWriter ← 历史报警 (history)
       └─ InMemoryAlarmWriter (开发/测试，内存)
```

**IAlarmWriter 接口** (`Shared/Interfaces/IAlarmWriter.cs`):

| 方法 | 职责 | 生产实现 |
|------|------|----------|
| `WriteRealtimeAlarmAsync` | 写入/更新实时报警 | Redis |
| `ClearRealtimeAlarmAsync` | 清除实时报警 | Redis |
| `WriteHistoryAlarmAsync` | 写入历史事件 | ClickHouse |
| `GetActiveAlarmsAsync` | 查询全部活跃报警 | Redis |
| `GetAlarmAsync` | 查询单个监视项的报警 | Redis |

## 二、计算循环 (WorkerCalculationService)

### 2.1 主循环 (`ExecuteAsync`)

```
while !cancelled:
    ① 收集所有监视项的 TagName + 规则中引用的 TagName
    ② batch read 从 TrendDB/SimulatedTrendReader 获取实时值
    ③ 按 RefreshIntervalSecond 筛选到期监视项
    ④ Parallel.ForEach → ProcessMonitorAsync
    sleep 100ms
```

### 2.2 ProcessMonitorAsync — 三步处理

```
ProcessMonitorAsync(monitor, values, now)
  │
  ├─ ① 前置规则检查 (PrerulePipeline.CheckAsync)
  │     ├─ ManualFlag: 手动停止标志
  │     ├─ StopMonitor: 关联启停监视项处于报警 → suppress
  │     └─ SourceDependency: 数据源依赖不可用 → suppress
  │
  │     若 ShouldSuppress:
  │       ├─ ShouldClearAlarm? → 写 clear 事件 + 清除实时报警 + 重置状态
  │       └─ return
  │
  ├─ ② 规则计算 (RuleDispatcher.CalculateAsync → 各 Calculator)
  │     └─ 返回 RuleCalculateResult { HasEvent, State, States[], TriggerValue }
  │
  └─ ③ 报警写入 (根据 HasEvent 分支)
        ├─ HasEvent=true  → [触发路径] 写实时报警 + 历史 trigger 事件
        └─ HasEvent=false → [消除路径] 若 PreviousStatus != null, 写 clear 事件
```

## 三、触发路径 (HasEvent = true)

### 3.1 流程

```
规则计算返回 HasEvent=true
  │
  ├─ 收集 alarmStates (支持多状态: State, States[], StatesDic)
  │
  ├─ GetAlarmAsync(monitor.Id) → 查询 Redis 中当前活跃报警
  │    └─ 生产环境: RedisAlarmWriter → Redis HGET ruleengine:alarm:{id} full_json
  │
  ├─ isNewAlarm = (existingAlarm == null || 状态不同)
  │
  ├─ foreach stateKey: WriteRealtimeAlarmAsync(alarm)
  │    └─ 生产环境: Redis HSET ruleengine:alarm:{id} ... + SADD ruleengine:active_alarms {id}
  │
  └─ if isNewAlarm:
       ├─ WriteHistoryAlarmAsync(triggerEvent)
       │    └─ 生产环境: ClickHouseAlarmWriter → Channel → batch INSERT
       │
       └─ 保存 state:
            state.PreviousStatus = firstState
            state.PreviousEventOccurTime = now
            SaveAsync(monitor.Id, state)
```

### 3.2 关键逻辑: isNewAlarm 判断

```csharp
var existingAlarm = await _alarmWriter.GetAlarmAsync(monitor.Id);
bool isNewAlarm = validStates.Count > 0
    && (existingAlarm == null || !validStates.Contains(existingAlarm.StatusKey));
```

- `existingAlarm` 查的是 Redis 中的实时报警 (`ruleengine:alarm:{monitorId}`)
- 产生新报警条件: Redis 中没有该监视项 OR Redis 中的 statusKey 与当前不同
- **目的**: 同一监视项在同一个报警状态下，只在首次满足条件时写一条 trigger 事件，不是每个周期都重复写

### 3.3 ClickHouse 写入细节

**trigger 事件的 ClickHouse 行:**
| 字段 | 值 | 来源 |
|------|-----|------|
| `event_type` | `'trigger'` | 固定 |
| `occur_time` | 当前 UTC 时间 | `now` |
| `clear_time` | `NULL` | trigger 事件无消除时间 |
| `trigger_value` | 触发值 | `result.TriggerValue` |
| `unit` | 单位 | `MonitorSources` 中 FocusSource 的 Unit |
| `job_id` | `{WorkerId}_{MonitorKey}_{yyyyMMddHHmmss}` | 自动生成 |

## 四、消除路径 (HasEvent = false + PreviousStatus != null)

### 4.1 流程

```
规则计算返回 HasEvent=false
  │
  ├─ 检查 state.PreviousStatus
  │    ├─ null → 无历史报警，跳过
  │    └─ != null → 需要消除!
  │
  └─ 消除处理:
       ├─ WriteHistoryAlarmAsync(clearEvent)
       │    │  clearEvent.RelatedTriggerOccurTime = state.PreviousEventOccurTime
       │    │
       │    └─ ClickHouseAlarmWriter:
       │         ├─ ① INSERT clear 行 (event_type='clear', clear_time=now)
       │         └─ ② ALTER TABLE UPDATE:
       │              UPDATE clear_time = now
       │              WHERE monitor_id = xx AND status_key = xx
       │                AND event_type = 'trigger'
       │                AND clear_time IS NULL
       │                AND occur_time = '{RelatedTriggerOccurTime}'
       │              SETTINGS mutations_sync = 1
       │
       ├─ ClearRealtimeAlarmAsync(monitor.Id)
       │    └─ Redis: DEL ruleengine:alarm:{id} + SREM ruleengine:active_alarms {id}
       │
       └─ 重置状态:
            state.PreviousStatus = null
            state.PreviousEventOccurTime = null
            SaveAsync(monitor.Id, state)
```

### 4.2 Clear 事件的两种 ClickHouse 操作

```
Step 1: INSERT — 写入一条独立的 clear 行
  event_type='clear', occur_time=消除时间, clear_time=消除时间,
  trigger_value→NULL, threshold_value=NULL

Step 2: ALTER TABLE UPDATE — 回填对应 trigger 行的 clear_time
  目的: HistoryMonitorAppService 查询 trigger 事件时通过 clear_time 计算 endTime/Duration
```

### 4.3 精确匹配策略

Clear 事件携带 `RelatedTriggerOccurTime`，在 ALTER TABLE UPDATE 时用于精确匹配:

```sql
ALTER TABLE ruleengine.alarm_events
UPDATE clear_time = '{clearEvent.OccurTime}'
WHERE monitor_id = '{monitorId}'
  AND status_key = '{statusKey}'
  AND event_type = 'trigger'
  AND clear_time IS NULL
  AND occur_time = '{RelatedTriggerOccurTime}'  -- 精确匹配，避免更新到错误行
SETTINGS mutations_sync = 1  -- 同步等待，确保写入立即可见
```

**RelatedTriggerOccurTime 的来源链路:**
1. trigger 时 → `state.PreviousEventOccurTime = now`
2. clear 时 → `clearEvent.RelatedTriggerOccurTime = state.PreviousEventOccurTime`
3. ClickHouse ALTER TABLE → `occur_time = RelatedTriggerOccurTime`

### 4.4 防止重复 Clear

Clear 后立即重置:
```csharp
state.PreviousStatus = null;
state.PreviousEventOccurTime = null;
await _stateStore.SaveAsync(monitor.Id, state);
```

下一周期再次 `HasEvent=false` 时，`state.PreviousStatus == null`，不会再写 clear 事件。

## 五、前置规则抑制的 Clear 逻辑

```csharp
if (preruleResult.ShouldSuppress)
{
    if (preruleResult.ShouldClearAlarm)
    {
        ClearRealtimeAlarmAsync(monitor.Id);  // Redis 清除

        // 读取报警状态
        var preruleState = await _stateStore.GetAsync(monitor.Id);
        if (preruleState?.PreviousStatus != null)
        {
            WriteHistoryAlarmAsync(clearEvent);  // ClickHouse 写 clear
            // 重置状态
            preruleState.PreviousStatus = null;
            preruleState.PreviousEventOccurTime = null;
            SaveAsync(monitor.Id, preruleState);
        }
    }
    return; // 跳过规则计算
}
```

三种前置规则产生 suppress 的场景:

| 前置规则 | ShouldSuppress | ShouldClearAlarm | 含义 |
|----------|:---:|:---:|------|
| ManualFlag | true | true | 手动标记停止，清除已有报警 |
| StopMonitor | true | true | 关联启停监视项告警，抑制并清除 |
| SourceDependency | true | false | 数据源不可用，仅抑制不消除 |

## 六、存储实现对比

### 6.1 Redis (实时报警)

Key 结构:
```
ruleengine:alarm:{monitorId}     → Hash (monitor_id, monitor_key, ... full_json)
ruleengine:active_alarms         → Set (monitorId 集合)
```

- TTL: 5 分钟保活 (每周期覆写时刷新)
- 作用: 实时报警列表查询 (`GET /api/ruleengine/alarms/realtime`)

### 6.2 ClickHouse (历史报警)

表: `ruleengine.alarm_events` (19 列)
- `event_type`: Enum8 (1=trigger, 2=clear)
- `occur_time`: DateTime64(3)
- `clear_time`: Nullable DateTime64(3) — trigger 行由后续 clear 事件的 ALTER TABLE UPDATE 回填
- 写入方式: 后台 Channel → 批量 INSERT (最多 500 条/批)
- Clear 事件额外执行: `ALTER TABLE ... UPDATE clear_time SETTINGS mutations_sync=1`

### 6.3 InMemoryStateStore (计算状态)

Key: `monitor.Id` 或 `{monitor.Id}:rdur`
Value: `CalculationState` (PreviousStatus, PreviousEventOccurTime, LastSatisfiedTimeMs, etc.)

两个独立 state key:
- `monitor.Id`: 报警生命周期状态 (`ProcessMonitorAsync` 使用)
- `{monitor.Id}:rdur`: 区间时长累计状态 (`CalculateRuleRangeDuration` 使用)

## 七、ClickHouseAlarmWriter 后台刷新

```
WriteHistoryAlarmAsync(alarmEvent)
  → Channel<AlarmEvent>.Writer.TryWrite(alarmEvent)
       ├─ 成功 → 返回
       └─ 失败 (channel 满) → DropOldest, Interlocked.Increment(_totalDropped)

后台 FlushLoop:
  while !cancelled:
    await channel.Reader.WaitToReadAsync()
    收集最多 500 条
    INSERT INTO alarm_events (...) VALUES (...), (...), ...
    foreach clearEvent:
      ALTER TABLE UPDATE clear_time (同步, mutations_sync=1)
```

Channel 容量: 10000, 满时策略: `DropOldest`

## 八、读取路径

### 实时报警

```
GET /api/ruleengine/alarms/realtime
  → AlarmQueryService.GetRealtimeAlarmsAsync()
    → IAlarmWriter.GetActiveAlarmsAsync()
      → Redis: SMEMBERS ruleengine:active_alarms → 遍历 HGET full_json
```

### 历史报警

```
POST /api/ruleengine/alarms/history
  → HistoryAlarmService.QueryAsync(request)
    → ClickHouse SQL:
        SELECT monitor_id, ..., event_type, occur_time, clear_time, ...
        FROM alarm_events
        WHERE ... (monitorKeys, startTime, endTime, eventTypes, statusKeys)
        ORDER BY occur_time DESC
        LIMIT x OFFSET y
```

**注意**: HistoryAlarmService 不做 trigger/clear 配对，只返回原始行。
配对逻辑在 MonitorCenter 的 `HistoryMonitorAppService.GetAllForListFromRuleEngineAsync`:
- 查询 trigger 事件 (`eventTypes=["trigger"]`)
- 跳过 clear 事件
- 用 `alarm.clearTime` 作为 `endTime`，如果 NULL 则 `endTime=null`

### 闭环验证

```
GET /api/ruleengine/alarms/history/closed-loop/validate
  → HistoryAlarmService.ValidateClosedLoopAsync(7天前, now)
    → ClickHouse GROUP BY:
        HAVING trigger_count > clear_count
```

## 九、已修复缺陷

| # | 缺陷 | 根因 | 修复 |
|---|------|------|------|
| D1 | 0 clear 事件 | `state` 首次为 null 时 `if (state != null)` 跳过，`PreviousStatus` 永不保存 | `state ??= new CalculationState { MonitorId = monitor.Id }` |
| D2 | 重复 clear | clear 后未重置 `PreviousStatus` | clear 后设 `PreviousStatus = null` |
| D6 | 虚假 clear (状态键碰撞) | `CalculateRuleRangeDuration` 和 `ProcessMonitorAsync` 共用 `monitor.Id` 作为 state key | 各用独立 key: `{id}:rdur` vs `{id}` |

## 十、Current State Key Layout

```
IStateStore (InMemoryStateStore / RocksDB):

  Key: "{monitor.Id}"           → CalculationState (alarm lifecycle)
      ├─ PreviousStatus         ← trigger 时写入, clear 后重置为 null
      ├─ PreviousEventOccurTime ← trigger 时写入, clear 后重置为 null  
      └─ (other fields unused by alarm path)

  Key: "{monitor.Id}:rdur"      → CalculationState (duration tracking)
      ├─ LastSatisfiedTimeMs
      ├─ AccumulatedDurationSec
      └─ PreviousStatus         ← duration 命中后的状态, 不用于 alarm clear 判断
```
