# 报警写入逻辑说明 (v2: 事件流模型)

## 一、架构总览

```
WorkerCalculationService (计算循环)
  └─ IAlarmWriter (报警写入接口)
       ├─ ProductionAlarmWriter (生产环境，复合)
       │    ├─ RedisAlarmWriter     ← 实时报警 (realtime)
       │    └─ ClickHouseAlarmWriter ← 历史事件 (history, INSERT-only)
       └─ InMemoryAlarmWriter (开发/测试，内存)
```

**IAlarmWriter 接口** (`Shared/Interfaces/IAlarmWriter.cs`):

| 方法 | 职责 | 生产实现 |
|------|------|----------|
| `WriteRealtimeAlarmAsync` | 写入/更新实时报警 | Redis |
| `ClearRealtimeAlarmAsync` | 清除实时报警 | Redis |
| `WriteHistoryAlarmAsync` | 写入状态变更事件 | ClickHouse |
| `GetActiveAlarmsAsync` | 查询全部活跃报警 | Redis |
| `GetAlarmAsync` | 查询单个监视项的报警 | Redis |

## 二、事件流模型 (v2)

### 2.1 核心原则

```
旧模型 (trigger/clear):
  触发 → INSERT trigger 行 (clear_time=NULL)
  消除 → INSERT clear 行 + ALTER TABLE UPDATE trigger 行
                              ↑ 阻塞，mutation 队列爆炸

新模型 (状态变更流):
  状态变化时 INSERT 一行: { monitor_id, status_key, occur_time, ... }
  无 trigger/clear 区分，无 ALTER TABLE UPDATE，纯追加写
  区间配对由查询侧 LEAD 窗口函数完成
```

### 2.2 ProcessMonitorAsync — 三步处理

```
ProcessMonitorAsync(monitor, values, now)
  │
  ├─ ① 前置规则检查 (PrerulePipeline.CheckAsync)
  │     ├─ ManualFlag: 手动停止标志
  │     ├─ StopMonitor: 关联启停监视项处于报警 → suppress
  │     └─ SourceDependency: 数据源依赖不可用 → suppress
  │
  │     若 ShouldSuppress:
  │       ├─ ShouldClearAlarm? → 写空状态事件 + 清除实时报警 + 设置 PreviousStatus=""
  │       └─ return
  │
  ├─ ② 规则计算 (RuleDispatcher.CalculateAsync → 各 Calculator)
  │     └─ 返回 RuleCalculateResult { HasEvent, State, States[], TriggerValue }
  │
  └─ ③ 事件写入 (根据状态变更)
        ├─ newStatus = HasEvent ? result.State : ""
        ├─ 实时报警: HasEvent → WriteRealtimeAlarm / !HasEvent → ClearRealtimeAlarm
        └─ 历史事件: newStatus != PreviousStatus → WriteHistoryAlarm + 更新 PreviousStatus
```

### 2.3 状态变更检测

```csharp
var newStatus = result.HasEvent ? result.State : "";
if (newStatus != state.PreviousStatus)
{
    // 写入状态变更事件
    WriteHistoryAlarmAsync(new AlarmEvent { StatusKey = newStatus, ... });
    state.PreviousStatus = newStatus;
    SaveAsync(monitor.Id, state);
}
```

| PreviousStatus | newStatus | 是否写入 | 含义 |
|:---:|:---:|:---:|------|
| null/"" | "satisfiled" | 是 | 首次触发报警 |
| "satisfiled" | "satisfiled" | 否 | 同状态保持，不写入 |
| "satisfiled" | "" | 是 | 报警恢复 |
| "" | "" | 否 | 持续正常，不写入 |
| "satisfiled" | "high_temp" | 是 | 状态切换 |

## 三、ClickHouse 写入

### 3.1 纯追加写

```sql
INSERT INTO ruleengine.alarm_events
(monitor_id, monitor_key, monitor_name, status_key, status_name,
 event_type, occur_time, clear_time, trigger_value, threshold_value,
 rule_type, config_version, worker_id, shard_id,
 last_event_id, last_event_name, unit, job_id)
VALUES (...)
```

- event_type 固定为 'trigger'（兼容旧列，不再区分 trigger/clear）
- clear_time 固定为 NULL（由 LEAD 窗口函数在查询侧计算）
- 无 ALTER TABLE UPDATE（移除阻塞 mutation）

### 3.2 Channel 批量写入

- Channel 容量: 10000
- 批量大小: 最多 500 条/批
- 满时策略: DropOldest
- 后台刷新: 异步循环，连接复用

## 四、查询侧配对 (LEAD 窗口函数)

### 4.1 HistoryAlarmService.QueryAsync

```sql
SELECT monitor_id, monitor_key, monitor_name, status_key, status_name,
       start_time, end_time, trigger_value, worker_id,
       last_event_id, last_event_name, unit, job_id
FROM (
    SELECT monitor_id, monitor_key, monitor_name, status_key, status_name,
           occur_time AS start_time,
           LEAD(occur_time) OVER (PARTITION BY monitor_id ORDER BY occur_time) AS end_time,
           trigger_value, worker_id,
           last_event_id, last_event_name, unit, job_id
    FROM ruleengine.alarm_events
    WHERE {where}
)
WHERE status_key != ''   ← 仅当 containNull=false 时生效
ORDER BY start_time DESC
LIMIT {limit} OFFSET {offset}
```

- 内部子查询: 对每个 monitor 的事件按时间排序，LEAD 取下一事件的 occur_time 作为 end_time
- 外部过滤: `status_key != ''` 只返回有效状态事件（空状态事件作为区间末端被消费）
- containNull=true 时跳过 `status_key != ''` 过滤，让空状态事件也出现在结果中

### 4.2 containNull 参数链路

```
前端 containNull=true
  → EvnetDataAppService / HistoryMonitorAppService
    → RuleEngineClient.QueryHistoryAlarmsAsync(containNull: true)
      → POST /api/ruleengine/alarms/history { containNull: true }
        → HistoryAlarmService.QueryAsync
          → SQL: 不添加 status_key != '' 过滤
```

### 4.3 空状态事件 (StatusKey = "")

**写入时机:**

| 场景 | 触发条件 | PreviousStatus → newStatus |
|------|------|------|
| 前置规则清除 | Prerule.ShouldSuppress && ShouldClearAlarm | "satisfiled" → "" |
| 规则不再满足 | HasEvent = false | "satisfiled" → "" |
| 规则首次不满足 | HasEvent = false, PreviousStatus = null | null → "" (首次不做变更) |

**查询行为:**

| containNull | SQL 过滤 | 返回结果 |
|:---:|------|------|
| false (默认) | `WHERE status_key != ''` | 仅有效状态事件 (报警区间) |
| true | 无此过滤 | 包含空状态事件 (显示恢复时刻) |

**前端影响:**

- **事件列表 (event-list)**: containNull=true 时显示清除事件 (StatusKey="")，前端展示为"恢复正常"
- **历史列表 (history-list)**: 空状态事件是 LEAD 区间配对的末端；containNull=true 时显示持续时长为 0 的"恢复"行

```sql
-- 查找最后一条事件状态非空的 monitor（未关闭的报警）
SELECT monitor_id, monitor_key, monitor_name, status_key, ...
FROM (
    SELECT *, LEAD(status_key) OVER (PARTITION BY monitor_id ORDER BY occur_time) AS next_status
    FROM alarm_events
    WHERE ...
)
WHERE status_key != ''
GROUP BY monitor_id, monitor_key, monitor_name, status_key
HAVING next_status IS NULL OR next_status != ''
```

## 五、实时报警 (Redis)

## 六、一致性保证

| 场景 | 旧模型 (trigger/clear) | 新模型 (状态变更流) |
|------|:---:|:---:|
| StateStore 丢失 | clear 不再生成 → trigger 孤悬 | 下一周期重写相同状态（幂等） |
| Worker 漂移 | trigger 已写但无 clear → 孤悬 | 新 Worker 从 ClickHouse 查最后事件恢复状态 |
| 毫秒级状态抖动 | trigger/clear 配对错乱 | 每条独立，查询侧 LEAD 自然配对 |
| Channel 丢事件 | 丢一个配对断裂 | 最多丢一条中间事件，下一个自动补上 |
| 重复 clear | PreviousStatus 未重置 | 状态变更检测天然防重 |

## 七、存储实现对比

### 7.1 ClickHouse (历史事件)

表: `ruleengine.alarm_events`
- 无 event_type 列（新 schema）或固定写 'trigger'（兼容旧 schema）
- 无 clear_time 列（新 schema）或固定写 NULL（兼容旧 schema）
- 写入方式: 后台 Channel → 批量 INSERT（最多 500 条/批）
- **不再执行 ALTER TABLE UPDATE**

### 7.2 Redis (实时报警)

与 v1 完全相同，无变化。

### 7.3 InMemoryStateStore (计算状态)

Key: `monitor.Id` 或 `{monitor.Id}:rdur`
Value: `CalculationState`
- `PreviousStatus`: 记录最后写入的状态键（"" 表示正常，非空表示报警中）
- 不再需要 `PreviousEventOccurTime`（v2 不需要精确匹配 clear）
