# 规则引擎（RuleEngine）数据库设计文档

## 1. 概述

### 1.1 存储架构

RuleEngine 不使用传统关系型数据库（RDBMS）。三种存储各司其职：

```
┌──────────────┬──────────┬──────────────────────────────┐
│  存储引擎     │  用途     │  说明                         │
├──────────────┼──────────┼──────────────────────────────┤
│ Redis        │ 实时报警  │ 当前活跃报警快照、Worker 状态、│
│              │          │ 分区映射表、Master 分布式锁      │
├──────────────┼──────────┼──────────────────────────────┤
│ ClickHouse   │ 历史报警  │ 所有报警事件（产生和消除）、    │
│              │          │ 按天分区、TTL 365 天自动清理     │
├──────────────┼──────────┼──────────────────────────────┤
│ RocksDB      │ 计算状态  │ Worker 本地的有状态计算持久化、 │
│              │          │ 每个 Worker 独立实例             │
└──────────────┴──────────┴──────────────────────────────┘
```

### 1.2 为什么不用 PostgreSQL

| 对比维度 | MonitorCenter (PostgreSQL) | RuleEngine |
|---------|---------------------------|------------|
| 实时查询 | SELECT * FROM ssmccureventdat | Redis HSET/HGET |
| 历史查询 | SELECT * FROM ssmceventdat WHERE ... | ClickHouse MergeTree |
| 计算状态 | 64-shard ConcurrentDictionary (内存) | RocksDB (持久化) |
| 查询延迟 | 10-100ms | Redis < 1ms / ClickHouse 10-50ms |
| 扩展性 | 垂直扩展 | 水平扩展 |
| 依赖 | 需要 PostgreSQL 服务 | 无需传统 RDBMS |

## 2. Redis 数据模型

### 2.1 实时报警快照

```
Key 模式: alarm:current:{monitor_id}
类型:     Hash
TTL:      无（报警消除时主动删除）
大小估算: ~500B/条 × 活跃报警数（通常 < 1% × 总监控项数）

示例:
  HSET alarm:current:guid-001
    monitorId      "guid-001"
    monitorKey     "temp_boiler_01"
    monitorName    "锅炉温度监控"
    statusKey      "high_temp_alarm"
    statusName     "超温报警"
    value          "105.3"
    occurTime      "2026-07-17T08:30:15.000Z"
    configVersion  "2026-07-17T00:00:00Z"
    workerId       "worker-3"
```

### 2.2 活跃报警索引

```
Key 模式: alarm:active
类型:     Set
TTL:      无
大小估算: 成员数 × ~40B

示例:
  SMEMBERS alarm:active
    → ["guid-001", "guid-005", "guid-012", ...]

查询流程:
  1. SMEMBERS alarm:active                 → 获取所有活跃报警 ID
  2. HMGET alarm:current:{id1} alarm:current:{id2} ... → 批量获取快照
```

### 2.3 分区映射表

```
Key 模式: shard:monitor_to_worker
类型:     Hash
TTL:      无（由 Master 维护，定期全量覆盖）

示例:
  HSET shard:monitor_to_worker
    guid-001  "worker-1"
    guid-002  "worker-3"
    guid-003  "worker-1"
    ...

用途:
  - Master 重启后恢复分区信息
  - 故障转移时查找宕机 Worker 的监控项
  - 外部系统查询某个监控项在哪个 Worker 上
```

### 2.4 Worker 心跳

```
Key 模式: worker:heartbeat:{worker_id}
类型:     String
TTL:      30 秒
Value:    Unix 时间戳（毫秒）

示例:
  SET worker:heartbeat:worker-3 "1752745815000" EX 30

用途:
  - Master 检测 Worker 存活状态
  - 30 秒内未续约 → Worker 被判死亡 → 触发故障转移
```

### 2.5 Master 分布式锁

```
Key 模式: master:leader
类型:     String
TTL:      15 秒
Value:    Master 实例 ID

获取锁:
  SET master:leader "master-instance-1" NX EX 15

续约 (Lua 脚本):
  if redis.call("GET", KEYS[1]) == ARGV[1] then
      return redis.call("EXPIRE", KEYS[1], ARGV[2])
  else
      return 0
  end

释放 (Lua 脚本):
  if redis.call("GET", KEYS[1]) == ARGV[1] then
      return redis.call("DEL", KEYS[1])
  else
      return 0
  end
```

### 2.6 配置同步版本号

```
Key 模式: config:version
类型:     String
TTL:      无
Value:    ISO 8601 时间戳

示例:
  SET config:version "2026-07-17T10:30:00.000Z"

用途:
  - MonitorCenter 推送配置变更时带来的版本号
  - Master 校验配置版本，防止过期更新
  - Worker 计算结果携带 configVersion，查询方可过滤过期数据
```

### 2.7 Redis 数据汇总

| Key 模式 | 类型 | TTL | 写入者 | 读取者 |
|---------|------|-----|--------|--------|
| `alarm:current:{id}` | Hash | 无（主动删） | Worker | Master (查询 API) |
| `alarm:active` | Set | 无 | Worker | Master |
| `shard:monitor_to_worker` | Hash | 无 | Master | Master, 外部查询 |
| `worker:heartbeat:{id}` | String | 30s | Worker | Master |
| `master:leader` | String | 15s | Master | Master |
| `config:version` | String | 无 | MonitorCenter | Master |

## 3. ClickHouse 数据模型

### 3.1 报警事件表

```sql
CREATE TABLE alarm_events (
    -- 主键维度
    monitor_id        String COMMENT '监视项唯一 ID',
    monitor_key       String COMMENT '监视项唯一编码',
    monitor_name      String COMMENT '监视项名称',

    -- 状态信息
    status_key        String COMMENT '状态键 (如 high_temp_alarm)',
    status_name       String COMMENT '状态名称 (如 超温报警)',
    event_type        Enum8('trigger' = 1, 'clear' = 2) COMMENT '事件类型: 触发/消除',

    -- 时间和数值
    occur_time        DateTime64(3) COMMENT '事件发生时间 (毫秒精度)',
    clear_time        Nullable(DateTime64(3)) COMMENT '报警消除时间',
    trigger_value     Float64 COMMENT '触发值',
    threshold_value   Nullable(Float64) COMMENT '阈值 (如有)',

    -- 元数据
    rule_type         UInt8 COMMENT '规则类型 (RuleType 枚举值)',
    config_version    DateTime64(3) COMMENT '规则配置版本号',
    worker_id         String COMMENT '产生此事件的 Worker ID',
    shard_id          UInt8 COMMENT '分片 ID',

    -- 分区键 (由 occur_time 派生)
    date              Date MATERIALIZED toDate(occur_time)
) ENGINE = MergeTree()
PARTITION BY toYYYYMMDD(occur_time)
ORDER BY (monitor_id, occur_time)
TTL occur_time + INTERVAL 365 DAY
SETTINGS index_granularity = 8192;
```

### 3.2 报警日统计物化视图

```sql
CREATE MATERIALIZED VIEW alarm_daily_stats_mv
ENGINE = SummingMergeTree()
PARTITION BY toYYYYMM(date)
ORDER BY (date, monitor_id, status_key)
AS SELECT
    toDate(occur_time) AS date,
    monitor_id,
    monitor_key,
    monitor_name,
    status_key,
    status_name,
    count() AS alarm_count,
    max(trigger_value) AS max_trigger_value,
    min(trigger_value) AS min_trigger_value,
    avg(trigger_value) AS avg_trigger_value
FROM alarm_events
WHERE event_type = 'trigger'
GROUP BY date, monitor_id, monitor_key, monitor_name, status_key, status_name;
```

### 3.3 常用查询

```sql
-- 查询指定时间范围的报警事件
SELECT monitor_id, monitor_name, status_key, status_name,
       occur_time, trigger_value, event_type
FROM alarm_events
WHERE monitor_id IN ('id1', 'id2')
  AND occur_time BETWEEN '2026-07-01' AND '2026-07-17'
ORDER BY occur_time DESC
LIMIT 100;

-- 按天统计报警数量
SELECT toDate(occur_time) AS day, count() AS cnt
FROM alarm_events
WHERE occur_time >= now() - INTERVAL 30 DAY
  AND event_type = 'trigger'
GROUP BY day
ORDER BY day;

-- 查询某监控项的报警时间线
SELECT occur_time, status_key, trigger_value, event_type
FROM alarm_events
WHERE monitor_id = 'guid-001'
  AND occur_time >= now() - INTERVAL 7 DAY
ORDER BY occur_time;

-- 查询当前仍在活跃的报警（通过 anti-join）
SELECT ae.*
FROM alarm_events ae
LEFT ANTI JOIN (
    SELECT DISTINCT monitor_id
    FROM alarm_events
    WHERE event_type = 'clear'
      AND occur_time >= now() - INTERVAL 7 DAY
) cleared ON ae.monitor_id = cleared.monitor_id
WHERE ae.event_type = 'trigger'
  AND ae.occur_time >= now() - INTERVAL 7 DAY;
```

### 3.4 ClickHouse 表汇总

| 表名 | 引擎 | 分区 | TTL | 说明 |
|------|------|------|-----|------|
| `alarm_events` | MergeTree | `toYYYYMMDD(occur_time)` | 365 天 | 原始报警事件 |
| `alarm_daily_stats_mv` | SummingMergeTree | `toYYYYMM(date)` | - | 按天聚合统计 |

## 4. RocksDB 数据模型

### 4.1 概述

RocksDB 是嵌入 Worker 进程的本地 KV 存储引擎（通过 `RocksDBSharp` NuGet 包），每个 Worker 有自己独立的实例。数据存储在 Worker 容器的本地卷中（`/data/rocksdb/`）。

### 4.2 Key/Value 设计

```
Key 格式:   state:{monitor_id}
Value 格式: JSON (序列化的 CalculationState)

示例:
  Key:   state:guid-001
  Value:
    {
      "monitorId": "guid-001",
      "ruleType": 2,
      "lastSatisfiedTimeMs": 1752745815000,
      "accumulatedDurationSec": 45.5,
      "frequencyWindow": [1752745000000, 1752745100000],
      "previousStatus": "high_temp_alarm",
      "configVersion": "2026-07-17T08:00:00Z"
    }

大小估算: ~200B/条 × 10万监控项/Worker ≈ 20MB
```

### 4.3 CalculationState JSON Schema

```json
{
    "$schema": "http://json-schema.org/draft-07/schema#",
    "type": "object",
    "properties": {
        "monitorId": { "type": "string" },
        "ruleType": { "type": "integer", "minimum": 0, "maximum": 9 },
        "lastSatisfiedTimeMs": { "type": "integer" },
        "accumulatedDurationSec": { "type": "number" },
        "frequencyWindow": {
            "type": "array",
            "items": { "type": "integer" }
        },
        "previousStatus": { "type": "string" },
        "configVersion": { "type": "string", "format": "date-time" }
    },
    "required": ["monitorId", "ruleType"]
}
```

### 4.4 按规则类型的状态字段

| 规则类型 | 存储的状态字段 |
|---------|--------------|
| Expression (1) | `previousStatus` |
| RangeDuration (2) | `lastSatisfiedTimeMs`, `accumulatedDurationSec`, `previousStatus` |
| RangeFrequency (3) | `frequencyWindow` (滑动窗口内触发时间戳列表) |
| FeatureValue (4) | `previousStatus` |
| PackageValue (5) | `previousStatus` (每个 bit 的状态) |
| WallTemperature (6) | `lastSatisfiedTimeMs`, `accumulatedDurationSec`, `previousStatus` |
| InterfaceMonitoring (7) | `previousStatus`, `lastSatisfiedTimeMs` |
| RulePackageValue (8) | `previousStatus` (多段打包状态) |
| RuleMultiStateRangeDuration (9) | `lastSatisfiedTimeMs` × N (每个子状态独立计数器) |

### 4.5 操作接口

```csharp
public class RocksDbStateStore : IStateStore
{
    private readonly RocksDb _db;
    private readonly ILogger<RocksDbStateStore> _logger;

    // 读取单个状态
    public async Task<CalculationState> GetAsync(string monitorId)
    {
        var key = Encoding.UTF8.GetBytes($"state:{monitorId}");
        var value = _db.Get(key);
        if (value == null) return null;
        return JsonSerializer.Deserialize<CalculationState>(value);
    }

    // 保存单个状态
    public async Task SaveAsync(string monitorId, CalculationState state)
    {
        var key = Encoding.UTF8.GetBytes($"state:{monitorId}");
        var value = JsonSerializer.SerializeToUtf8Bytes(state);
        _db.Put(key, value);
    }

    // 批量读取（WriteBatch 优化）
    public async Task<Dictionary<string, CalculationState>> GetBatchAsync(
        IEnumerable<string> monitorIds)
    {
        var result = new Dictionary<string, CalculationState>();
        foreach (var id in monitorIds)
        {
            var state = await GetAsync(id);
            if (state != null) result[id] = state;
        }
        return result;
    }

    // 批量保存
    public async Task SaveBatchAsync(Dictionary<string, CalculationState> states)
    {
        using var batch = new WriteBatch();
        foreach (var (id, state) in states)
        {
            var key = Encoding.UTF8.GetBytes($"state:{id}");
            var value = JsonSerializer.SerializeToUtf8Bytes(state);
            batch.Put(key, value);
        }
        _db.Write(batch);
    }

    // 删除状态
    public async Task DeleteAsync(string monitorId)
    {
        var key = Encoding.UTF8.GetBytes($"state:{monitorId}");
        _db.Remove(key);
    }
}
```

### 4.6 RocksDB 配置

```csharp
var options = new DbOptions()
    .SetCreateIfMissing(true)
    .SetCreateMissingColumnFamilies(true)
    .SetMaxBackgroundJobs(4)
    .SetBytesPerSync(1048576)       // 1MB
    .SetMaxBytesForLevelBase(268435456) // 256MB (Level 1 大小)
    .SetWriteBufferSize(67108864)   // 64MB (MemTable 大小)
    .SetMaxWriteBufferNumber(3)
    .SetMinWriteBufferNumberToMerge(1)
    .SetCompression(CompressionTypeEnum.lz4);
```

### 4.7 状态迁移协议

当监控项从 Worker-A 迁移到 Worker-B 时：

```
1. Master 通知 Worker-A: "准备迁移 monitorId=guid-001"
2. Worker-A:
   a. 停止计算 guid-001
   b. 读取 RocksDB: GET state:guid-001
   c. 序列化状态 + 最新计算结果
   d. gRPC 发送给 Worker-B
3. Worker-B:
   a. 接收状态快照
   b. 写入 RocksDB: PUT state:guid-001
   c. 初始化本地规则缓存
   d. 通知 Master: "guid-001 已就绪"
4. Master 更新分区映射表: HSET shard:monitor_to_worker guid-001 worker-B
5. Master 通知 Worker-A: 删除 guid-001 的本地数据
```

### 4.8 RocksDB 表汇总

| Key 前缀 | Value 格式 | 大小 | 说明 |
|---------|-----------|------|------|
| `state:{id}` | JSON | ~200B/条 | 计算中间状态 |
| 总计 (10万项/Worker) | - | ~20MB | 非常轻量 |

## 5. 存储容量规划

### 5.1 100 万监控项规模

| 存储 | 数据类型 | 单条大小 | 数量 | 总容量 |
|------|---------|---------|------|--------|
| Redis | 实时报警快照 | 500B | ~10,000 (1% 活跃) | ~5MB |
| Redis | Worker 心跳 | 50B | ~100 (Worker 数量) | ~5KB |
| Redis | 分区映射表 | 80B | 1,000,000 | ~80MB |
| Redis | 其他 (锁/版本号) | 100B | ~10 | ~1KB |
| **Redis 总计** | | | | **~85MB** |
| ClickHouse | 报警事件 | 300B | ~50万/天 (假设 5% 报警率) | ~150MB/天 |
| **ClickHouse 年计** | | | | **~55GB** (压缩后 ~15GB) |
| RocksDB | 计算状态 | 200B | 100,000/Worker | ~20MB/Worker |

### 5.2 资源分配建议

| 组件 | 内存 | 磁盘 | 说明 |
|------|------|------|------|
| Redis | 512MB | 5GB | 内存远大于数据量，留足余量 |
| ClickHouse | 2GB | 200GB | 压缩比约 3:1，实际占用小 |
| RocksDB (每 Worker) | 128MB (Block Cache) | 5GB | 嵌入进程内，无需独立资源 |
