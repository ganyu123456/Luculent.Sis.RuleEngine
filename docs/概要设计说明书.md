# 规则引擎（RuleEngine）概要设计说明书

## 1. 引言

### 1.1 项目概述

规则引擎（RuleEngine）是 LiEMS SIS 平台的分布式规则计算服务，从现有 MonitorCenter 单体管道架构演进而来。系统采用 Control Plane / Data Plane 分离架构，Master 负责配置管理与分区调度，Worker 集群负责时序数据采集与规则计算，计算结果写入 Redis（实时报警）和 ClickHouse（历史报警）。

### 1.2 技术栈

| 层次 | 技术选型 | 说明 |
|------|----------|------|
| 运行时 | .NET 10.0 | 与 MonitorCenter (.NET 6) 完全解耦，零项目引用 |
| Worker 编译 | Native AOT | 启动 < 0.3s，内存 ~50MB，容器镜像 ~40MB |
| Master 编译 | JIT (默认) | 保留反射能力，支持动态配置管理 |
| 通信 | gRPC (内部) / HTTP REST (对外) | Worker ↔ Master 用 gRPC，对外查询用 HTTP |
| 任务编排 | TPL Dataflow | Worker 内部管线，复用 MonitorCenter 的管道模式 |
| 对象映射 | 手动映射 | 不引入 AutoMapper，保持依赖极简 |
| 日志 | Microsoft.Extensions.Logging | 标准 .NET 日志抽象 |
| 配置 | Microsoft.Extensions.Configuration | JSON + 环境变量 |
| 实时存储 | Redis (StackExchange.Redis) | 当前报警快照 + Master 选主 + 分区映射表 |
| 历史存储 | ClickHouse (ClickHouse.Client) | 列式存储，海量报警事件 |
| 本地状态 | RocksDB (RocksDBSharp) | Worker 本地有状态计算的持久化 |
| 时序数据 | TrendDB DLL 直读 | 托管程序集，内部 gRPC 通信，已验证 .NET 10 兼容 |
| 容器化 | Docker + Docker Compose | linux-amd64，chiseled 镜像 |

### 1.3 与 MonitorCenter 的关系

```
MonitorCenter (保留)                 RuleEngine (新增)
  ├── 监控项管理 UI                   ├── 规则计算引擎 (Worker 集群)
  ├── 监控树/状态/数据源管理            ├── 实时报警存储 (Redis)
  ├── 历史数据查询 (逐步迁移)           ├── 历史报警存储 (ClickHouse)
  ├── 重算管理                         ├── 对外查询 API (Master HTTP)
  └── 配置变更通知 ───HTTP──►          └── 接收配置变更 → 分发给 Worker
```

**硬性约束：RuleEngine 不引用 MonitorCenter 任何项目或类库。** 交互仅通过 HTTP API（配置同步）和 Redis（报警查询），两边各自维护独立的类定义。

### 1.4 核心设计决策

| 决策 | 说明 |
|------|------|
| Control Plane / Data Plane 分离 | Master 管配置和分区，Worker 管计算，职责清晰 |
| Cost-Aware 静态分区 | 主路由策略，贪心装箱，各 Worker 负载差异 < 5% |
| 一致性哈希 + 有界负载 | 仅用于 Worker 宕机应急接管，秒级恢复 |
| 配置走本地内存 | Worker 热路径零网络开销，变更时 Master 增量推送 |
| 实时报警走 Redis | 分布式共享内存，多 Master 天然支持 |
| 历史报警走 ClickHouse | 列式存储，按天分区，TTL 365 天自动清理 |
| Worker Native AOT | 启动快、内存小、容器镜像小 |
| .NET 10.0 独立项目 | 与 MonitorCenter 零依赖，独立部署升级 |

## 4. 分片与负载均衡

### 4.1 双层策略

```
第一层 — Cost-Aware Static Partitioning (稳态，99% 时间)
  cost = ComplexityWeight[RuleType] × (1.0 / RefreshIntervalSecond)
  贪心装箱: 按 cost 降序排列 → 依次分配给当前最轻 Worker
  结果: 各 Worker 负载差异 < 5%

第二层 — Consistent Hashing with Bounded Loads (瞬态，1% 时间)
  Worker 宕机时:
    → 宕机 Worker 的监控项沿哈希环找下一个可用 Worker
    → 每个 Worker 负载上限 capacity = avg × (1 + ε)，ε = 0.2
    → 1 秒内恢复计算
```

### 4.2 复杂度权重

| 规则类型 | 权重 | 原因 |
|---------|------|------|
| Expression | 1.0 | 纯数学运算 |
| RangeDuration | 2.0 | 需查极值缓存 |
| RangeFrequency | 3.0 | 需遍历频率窗口 |
| PackageValue | 2.0 | 位解析 |
| FeatureValue | 2.0 | 特征值匹配 |
| RulePackageValue | 3.0 | 多段位打包 |
| RuleMultiStateRangeDuration | 4.0 | 多状态并发 |
| WallTemperature | 5.0 | 多变量关联 + IO |
| InterfaceMonitoring | 5.0 | 外部 HTTP 调用 |

## 5. 容错与高可用

### 5.1 故障场景

| 故障 | 影响 | 应对 | 恢复时间 |
|------|------|------|---------|
| Worker 宕机 | 该分片停止计算 | 哈希环自动接管 → Master 异步重平衡 | < 1s / < 5min |
| Master 宕机 | 配置变更、查询不可用 | Redis 选主，Standby 接管 | < 15s |
| Redis 宕机 | 实时报警查询不可用 | Redis Sentinel | < 30s |
| ClickHouse 宕机 | 历史报警写入积压 | Worker 本地缓冲，恢复后补写 | < 5min |
| TrendDB 不可用 | 所有 Worker 取不到数据 | 重试 + 告警 | 依赖 TrendDB |

### 5.2 Master 高可用

- 多 Master 实例，通过 Redis 分布式锁选举 Leader
- 锁 TTL 15s，Leader 每 5s 续约
- 分区映射表持久化在 Redis，新 Leader 可直接读取
- 全量配置从 MonitorCenter 重新拉取

## 6. 部署架构

### 6.1 服务组件

| 组件 | 编译方式 | 镜像 | 端口 |
|------|---------|------|------|

### 6.3 规模参考

| 监控项 | Worker | Master | 说明 |
