## 二、异常场景测试验收

> 详细测试步骤见 `docs/异常场景测试手册.md`

规则引擎作为生产环境核心组件，必须验证以下异常场景的自动恢复能力。每次发版前需执行完整回归测试。

### 2.1 测试场景清单

| 场景 | 操作 | 核心验收标准 |
|------|------|-------------|
| 场景 1: Worker 重启 | `docker restart` 单个 Worker | Worker 30s 内恢复，事件继续写入，lastEventName 恢复 |
| 场景 1b: 全部 Worker 重启 | 同时重启所有 Worker | 60s 内全部恢复，无死 Worker 误判 |
| 场景 2: Master 重启 | `docker restart ruleengine-master` | API < 15s 恢复，Worker 60s 内重连，事件无中断 |
| 场景 2b: Master 长时间宕机 | `docker stop` Master 2 分钟 | 宕机期间 Worker 继续计算，恢复后重连正常 |
| 场景 3: SIS 服务重启 | `docker restart sis-service` | MonitorCenter API 恢复 < 30s，EvnetData/History 查询正常 |
| 场景 4: Worker 扩容 | `--scale worker=2→3` | 新 Worker 30s 内注册，重分区触发，监视项均匀分配 |
| 场景 4b: Worker 缩容 | `--scale worker=3→2` | 死 Worker 30s 检测，45s 内重分区，监视项总数不变 |
| 场景 5: ClickHouse 重启 | `docker restart` ClickHouse | 恢复 < 10s，Worker 写入恢复，事件总数不减少 |
| 场景 5b: Redis 重启 | `docker restart` Redis | 实时报警重建，核心计算不受影响 |
| 场景 6: 配置变更 | MonitorCenter 推送变更 | 健康检查正常，实时报警无异常 |

### 6.4 启动恢复流程

```
Worker 启动
  ├─ ① 注册到 Master
  ├─ ② 从本地 StateStore 批量加载状态
  └─ ③ 本地缺失 → ClickHouse 批量查最后事件:
       SELECT monitor_id, status_key
       FROM ruleengine.alarm_events
       WHERE monitor_id IN ({ids})
       ORDER BY occur_time DESC
       LIMIT 1 BY monitor_id
```

实现位置: `WorkerCalculationService.RunOneCycleAsync` (首次周期执行)
接口方法: `IAlarmWriter.GetLastEventStatusesAsync`
实现类: `ClickHouseAlarmWriter` (生产), `InMemoryAlarmWriter`/`RedisAlarmWriter` (返回空字典)

| 场景 | 恢复行为 |
|------|------|
| 正常重启 (StateStore 持久) | StateStore 有状态 → 直接使用 |
| Worker 漂移/首次启动 | StateStore 无状态 → ClickHouse 查询恢复 |
| 新监视项 (无历史事件) | ClickHouse 查询返回空 → PreviousStatus = "" |

## 八、数据准备验收

### 7.1 核心设计要求

**统一数据源：** 所有监视项和前置规则共用 TrendDB 测点 `db05.test1`，这是当前环境唯一具备实时值的测点。

**数据流：**
```
db05.test1 (TrendDB)
  → DataAcquisitionService (每秒采集)
  → TagValueStore (缓存实时值)
  → WorkerCalculationService (每秒计算)
     ├─ 前置规则评估: db05.test1 > prerule_threshold
     └─ 监视项计算:  db05.test1 > monitor_threshold (持续 N 秒)
```

**设计原则：**
- 所有监视项的 RealDB 数据源 (`source_flag=3`) 均指向 `db05.test1`
- 每个监视项通过不同的 Static 阈值 (`source_flag=1`) 实现差异化触发条件
- 每个监视项可配置不同 `duration_cnt` 实现错峰触发
- 前置规则也使用 `db05.test1`，在不同阈值条件下过滤监视项

### 7.2 监视项规模

| 指标 | 值 |
|------|-----|
| 总监视项数 | **≥ 10,000** |
| 前置规则数 | 2 (Expression A + RangeDuration B) |
| 前置规则分配 | 各关联约 5,000 个监视项 |
| 每个监视项数据源数 | 2 (1 RealDB + 1 Static) |
| 每个监视项规则数 | 1 (RangeDuration) |

### 7.4 数据完整性验证

- [ ] 2 个数据源记录 (`ssmcsourcemst`): 1 个 RealDB 指向 `db05.test1` + 1 个 Static 阈值
- [ ] 1 个区间时长规则 (`ssmcrulerandurmst`)
- [ ] RuleType = 2 (RangeDuration)
- [ ] `prerule_no` 指向 TEST-PRERULE-A 或 TEST-PRERULE-B

```bash
# 检查每个监视项的数据源和规则是否完整
docker exec sis-postgres psql -U postgres -d sis1 -c "
SELECT i.monitor_no,
       (SELECT COUNT(*) FROM ssmcsourcemst s WHERE s.group_no = i.monitor_no AND s.valid_sta = 'A') as src_count,
       (SELECT COUNT(*) FROM ssmcrulerandurmst r WHERE r.related_no = i.monitor_no AND r.valid_sta = 'A') as rule_count
FROM ssmcitemmst i
WHERE i.enable_flag = true
AND (
    (SELECT COUNT(*) FROM ssmcsourcemst s WHERE s.group_no = i.monitor_no AND s.valid_sta = 'A') != 2
    OR (SELECT COUNT(*) FROM ssmcrulerandurmst r WHERE r.related_no = i.monitor_no AND r.valid_sta = 'A') != 1
);"
# 预期: 0 行 (无不完整监视项)
```

### 7.5 关键配置要求 (重要!)

**org_no 一致性：** `ssmcitemmst.org_no`、`ssmcsourcemst.org_no`、`ssmcrulerandurmst.org_no` 必须一致（统一使用 `-1`）。MonitorCenter 的 ABP 多租户过滤器 (`IMayHaveTenant`) 按 `org_no` 过滤，不一致会导致 `GetAllMonitors()` 查不到数据源和规则。

**表关联关系：**
```
ssmcitemmst.monitor_no ← ssmcsourcemst.group_no       (数据源关联)
ssmcitemmst.monitor_no ← ssmcrulerandurmst.related_no  (规则关联)
ssmcitemmst.source_no  → ssmcsourcemst.source_no       (关注数据源，为空字符串表示无特定关注)
```

**数据源要求 (每个监视项 2 个)：**

| source_flag | source_id | source_cod | 说明 |
|-------------|-----------|------------|------|
| 3 (TrendDB) | `tag_value` | `db05.test1` | 实时趋势测点 (统一) |
| 1 (Static) | `threshold` | `30` ~ `90` | 阈值常量 (每个监视项不同，实现差异化触发) |

**规则要求 (每个监视项 1 个 RangeDuration)：**

| 字段 | 值 |
|------|-----|
| `left_id` | `tag_value` (匹配 TrendDB 数据源 source_id) |
| `right_id` | `threshold` (匹配阈值数据源 source_id) |
| `symbol_flag` | 1 (大于) |
| `duration_cnt` | 1 ~ 60 (秒，每个监视项不同，实现错峰触发) |
| `statuslin_cod` | `satisfiled` |
| `status_no` | `88a2ff2f827f487e83bda61736ca3ad1` (考核条件主题) |

**前置规则数据源要求 (每个前置规则 2 个)：**

| 前置规则 | source_flag | source_id | source_cod | 说明 |

## 十、已修复缺陷清单

| # | 缺陷 | 修复方式 |
|---|------|----------|
| D1 | Worker 漂移后 PreviousStatus 丢失导致重复事件 | Worker 启动恢复: StateStore 缺失时从 ClickHouse 查询最后事件恢复状态 |
| D2 | 事件流模型: ALTER TABLE UPDATE 阻塞 | 移除 ALTER TABLE UPDATE，改用 LEAD 窗口函数查询侧配对 |
| D7 | containNull=true 时不返回空状态事件 | HistoryAlarmService SQL 去掉硬编码 `status_key != ''`，改为根据 containNull 参数动态添加 |
| A3 | RelatedStatusName 显示事件状态名 | 分布式模式从 MonitorStatus 表查询 Name |
| A4 | OccurMonitorStatusName 为空 | 分布式模式从 MonitorStatusDefinition 表查询 Name |
| A5 | totalCount 与实际不一致 | 传递 `monitorKeys` 到 Rule Engine 查询 |
| B1 | 持续时长可能为负数 | MonitorCenter 侧负值守卫 (`endTimeMs < startTimeMs` → null) |
| B3 | totalCount 与实际不一致 | 传递 `monitorKeys` 到 Rule Engine 查询 |
| C1 | 本地模式分页未生效 | `GetAllForList` 和 `GetAllKeysForList` 添加 Skip/Take |
| A1/A2 | lastEventName/unit 等字段为空 | ClickHouse schema + Model + DTO + Writer + Service 全链路添加新字段 |
| F1 | RangeDuration BreakOnHit=false 始终返回 false | `PreruleEvaluationService` 添加 `anyHit` 标志 |
| F2 | 前置规则评估无日志 | `EvaluateAllAsync` 添加 per-rule 结果日志 |
| F3 | 使用 Key 而非 SourceKey 获取 RealDB 数据 | 分离 Static(SourceType=1) 和 RealDB(SourceType=3) 处理 |
| F4 | Redis 实时报警重复写入 | 状态变更判断内部才写入 Redis |
| F5 | 前置规则 suppress 路径重复清除报警 | `ClearRealtimeAlarmAsync` 移入 PreviousStatus 非空检查内部 |
| F6 | MonitorCenter 全表扫描 | 分页+JOIN 查询，Master 分批拉取 |
| G1 | MonitorCenter GetAllPrerules API 404 | `PreruleDatabaseReader` Npgsql 直读 PostgreSQL fallback |
| G2 | Admin API load-prerules 未推送到 Workers | 注入分区服务，加载后调用 `PushToWorkersAsync` |
| G3 | gRPC 重连日志噪音 | HTTP/2 reset 场景降级为 `LogInformation` |
