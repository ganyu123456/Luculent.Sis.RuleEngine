# 致命性能问题清单

> 目标规模: 100K 监视项/Worker，多规则类型混合
> 当前代码在此规模下无法正常运行，以下问题必须修复。

---

## F1. CalculateRuleExpression — 对全部 tag 做正则替换

**文件**: `Worker/Calculation/Calculators/CalculateRuleExpression.cs:28-35`

**现象**: `data` 字典来自 `TagValueStore.Values`，包含当前 Worker 分配到的**所有** tag（100K 条）。每条 Expression 规则计算时，都遍历整个字典做正则替换：

```csharp
foreach (var (tag, value) in data)        // ← 100K 次迭代
{
    var pattern = $@"\b{Regex.Escape(tag)}\b";
    expression = Regex.Replace(expression, pattern, value);
}
```

**量级**: 500 个 Expression 到期 × 100K tag × 每 tag 一次正则 = **5000 万次正则操作/周期**

**后果**: 单条 Expression 计算耗时 >100ms，1s 计算周期永远完不成，CPU 100%。

**修复**: 只替换表达式中实际出现的变量名，不遍历全部 tag。

```csharp
// 正确做法: 先解析表达式中的变量名，只替换这些变量
var usedTags = TagRegex().Matches(script)
    .Select(m => m.Value)
    .Distinct()
    .ToList();

foreach (var tag in usedTags)
{
    if (data.TryGetValue(tag, out var val) && val.HasValue)
        expression = expression.Replace(tag, val.Value.ToString(CultureInfo.InvariantCulture));
}
```

**预估收益**: 表达式计算从 100ms → <0.1ms，**1000x 加速**。

## F2. CalculateRuleExpression — 每次 new DataTable()

**文件**: `Worker/Calculation/Calculators/CalculateRuleExpression.cs:40-41`

**现象**: 每次求值创建一个 `DataTable` 对象，内部包含 schema、列集合、表达式编译器：

```csharp
var dt = new System.Data.DataTable();
var evaluated = dt.Compute(expression, null);
```

**量级**: 100K Expression × 刷新周期 60s = 每秒 ~1,667 次 `new DataTable()`

**后果**: 高频大对象分配 → gen2 GC 频繁触发 → GC 暂停 >100ms → 周期延迟雪崩。

**修复**: 用轻量表达式解析替代 `DataTable.Compute`。表达式格式简单（`a > b`, `a && b` 等），手动解析即可，已有 `PreruleEvaluationService.SafeEvaluate()` 可直接复用：

```csharp
// 参考 PreruleEvaluationService.SafeEvaluate 的实现
// 支持: > >= < <= == != && ||
// 用 double.Parse + switch 替代 DataTable.Compute
```

**预估收益**: 消除每次表达式求值的 gen0/gen2 分配，GC 暂停减少 80%+。

## F3. WorkerCalculationService — 计算与 I/O 混杂在 Parallel.ForEachAsync

**文件**: `Worker/WorkerCalculationService.cs:167-173`

**现象**: `MaxDegreeOfParallelism = Environment.ProcessorCount`（约 16 并发），每个 slot 内的 `ProcessMonitorAsync` 同步执行：

```
Redis GET (StopMonitor 检查) → 规则计算 → Redis HSET + SADD → ClickHouse channel
```

**量级**: 1,600+ 到期/周期 × 每监视项 ~3 I/O 操作 × 16 并发 = **槽位大部分时间在等网络**

**后果**: 
- 1s 周期内完不成所有到期监视项 → `_cycleLock` 下一周期被跳过 → 持续积压
- 短刷新间隔（1s/5s）的监视项永远得不到计算

**修复 A（小改动）** — 拆分为两阶段：

```csharp
// Phase 1: 纯计算 (CPU 密集，Parallel.ForEachAsync)
var results = new ConcurrentBag<(MonitorConfig, RuleCalculateResult, ...)>();
await Parallel.ForEachAsync(dueMonitors, ...);

// Phase 2: 批量写 I/O
foreach (var batch in results.Chunk(500))
{
    var redisTasks = batch.Select(...);      // 批量 Redis
    var clickhouseTasks = batch.Select(...);  // 批量 ClickHouse
    await Task.WhenAll(...);
}
```

**修复 B（理想）** — 实时报警也走 Channel，跟 ClickHouse 一样异步攒批写。

**预估收益**: 1s 周期可完成（16 并发纯计算 × 0.05ms × 1600 = 5ms），I/O 异步批量无阻塞。

## F4. GetAllTagNames — 每秒遍历 10 万监视项

**文件**: `Worker/WorkerCalculationService.cs:59-88`

**现象**: `DataAcquisitionService` 每秒调用一次 `GetAllTagNames()`，遍历全部监视项提取 tag：

```csharp
public HashSet<string> GetAllTagNames()
{
    foreach (var m in AssignedMonitors.Values)  // 10 万个
    {
        tags.Add(m.TagName);
        foreach (var r in ruleOpts.RangeDurationRules) { ... }   // 嵌套
        foreach (var r in ruleOpts.RangeFrequencyRules) { ... }  // 嵌套
        // ... WallTemperature 等
    }
}
```

**量级**: 10 万监视项 × 嵌套遍历配置字段 × 每秒 1 次 = **持续 CPU 消耗**

**后果**: 采集服务 CPU 占用高，与计算服务争抢 CPU，1s 采集周期可能延迟。

**修复**: 缓存 tag 列表，只在监视项分配变更时重建。

```csharp
private HashSet<string>? _cachedTagNames;

public HashSet<string> GetAllTagNames()
{
    // AssignedMonitors 变更时由 Master gRPC 推送触发缓存失效
    if (_cachedTagNames == null)
        _cachedTagNames = BuildTagNames();
    return _cachedTagNames;
}
```

**预估收益**: CPU 开销从持续高负载 → 仅变更时一次性重建。

## 优先级

| 编号 | 问题 | 严重程度 | 修复难度 |
|------|------|---------|---------|
| F1 | 表达式全量 tag 正则 | **致命** — 单条百 ms | 低 |
| F2 | DataTable 创建 | **严重** — GC 风暴 | 低 |
| F3 | 计算与 I/O 混杂 | **严重** — 周期超时 | 中 |
| F4 | GetAllTagNames 缓存 | **中等** — CPU 浪费 | 低 |

**建议顺序**: F1 → F2 → F3 → F4。F1+F2 修完 Expression 应该就能跑通，F3+F4 保稳定。
