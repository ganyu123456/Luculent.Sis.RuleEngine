# SIS 规则引擎（RuleEngine）架构设计文档

> 基于对现有 `00-Luculent.Sis.MonitorCenter` 项目代码的完整分析，结合分布式规则计算引擎的最佳实践编写。

---

## 1. 文档概述

### 1.1 项目定位

规则引擎（RuleEngine）是 LiEMS SIS 平台的下一代规则计算服务，从现有的 MonitorCenter 单体管道架构演进为独立部署的分布式计算引擎。核心目标：

- **解耦**：将规则计算从 SIS.Web Host 进程中独立出来
- **扩展**：支持多 Worker 水平扩展，突破单机 CPU 瓶颈
- **加速**：直连 TrendDB 获取时序数据，消除 gRPC → RealDB 中间链路
- **分层存储**：实时报警走 Redis，历史报警走 ClickHouse，各司其职

### 1.2 技术选型

| 层次 | 技术 | 说明 |
|------|------|------|
| 运行时 | .NET 10.0 | 最新 LTS，Native AOT 支持，与现有技术栈（.NET 6）完全解耦 |
| Worker 编译 | Native AOT | Worker 独立编译为原生二进制，启动 < 0.3s，内存 < 50MB |
| Master 编译 | JIT (默认) | Master 需要动态配置管理，保留完整反射能力 |
| 通信协议 | gRPC (Worker ↔ Master) / HTTP REST (对外 API) | 内部高性能 + 外部易集成 |
| 任务编排 | TPL Dataflow | 复用 MonitorCenter 的管道模式，Worker 内部管线 |
| 实时存储 | Redis | 当前报警快照，分布式共享内存 |
| 历史存储 | ClickHouse | 列式存储，海量报警事件查询 |
| 本地状态 | RocksDB | Worker 本地有状态计算的持久化 |
| 时序数据 | TrendDB DLL 直读 | 托管程序集，内部 gRPC 通信，已验证 .NET 10 兼容 |

### 1.3 Microsoft.Extensions 依赖

Worker 的依赖极简——不需要 ABP、不需要 EF Core、不需要 ASP.NET Core 完整栈：

```
RuleEngine.Worker
  ├── TrendDb_API.dll (TrendDB5 客户端，托管程序集)
  ├── Grpc.Net.Client 2.57+ (TrendDB 依赖)
  ├── StackExchange.Redis (实时报警写入)
  ├── ClickHouse.Client (历史报警写入)
  ├── RocksDBSharp (本地状态持久化)
  └── Microsoft.Extensions.Hosting (Worker 生命周期)
```

Master 则引入 ASP.NET Core Minimal API + gRPC 服务端：

```
RuleEngine.Master
  ├── ASP.NET Core (Minimal API，对外 HTTP)
  ├── Grpc.AspNetCore (Worker 通信)
  ├── StackExchange.Redis (实时报警查询 + 选主)
  ├── ClickHouse.Client (历史报警查询)
  └── Microsoft.Extensions.Hosting
```

### 1.4 与 MonitorCenter 的关系

```
MonitorCenter (保留)              RuleEngine (新增)
├── 监控项管理 UI                 ├── 规则计算引擎
├── 监控树/状态/数据源配置         ├── 实时报警存储 (Redis)
├── 历史数据查询 (逐步迁移)        ├── 历史报警存储 (ClickHouse)
├── 重算管理                      ├── 对外查询 API
└── 配置变更通知 →                └── 接收配置变更，分发给 Worker
```

MonitorCenter 保留管理职能（配置监控项、规则、状态），RuleEngine 接管计算职能（拿数据、算规则、产报警）。

### 1.4 项目依赖原则：零依赖，纯接口交互

**RuleEngine 不引用 MonitorCenter 的任何项目或类库。** 这是一个硬性约束，不是技术妥协。

| 维度 | 说明 |
|------|------|
| **项目引用** | RuleEngine 不引用 `Luculent.Sis.MonitorCenter.*` 任何项目 |
| **NuGet 依赖** | RuleEngine 不依赖 ABP 框架（MonitorCenter 的核心框架） |
| **数据库访问** | RuleEngine 不连接 MonitorCenter 的 PostgreSQL 数据库 |
| **实体/POCO** | RuleEngine 有自己的 POCO 定义，不共享 MonitorCenter 的 Entity/CacheItem |
| **计算逻辑** | 拷贝改造，不复用继承——MonitorCenter 原版不动，RuleEngine 有自己的计算层 |

**为什么要零依赖？**

```
依赖 MonitorCenter 项目的后果:
  ├── 被迫引入 ABP 框架（MonitorCenterBase、ITransientDependency 等）
  ├── 被迫引入 EF Core（MonitorCenterDbContext）
  ├── 被迫引入 MonitorCenter 的全部传递依赖（SIS 平台框架等）
  ├── 部署时必须带上 MonitorCenter 的 DLL → 和 SIS.Web Host 纠缠
  └── MonitorCenter 改一个 POCO 字段，RuleEngine 就得重新编译

零依赖的结果:
  ├── RuleEngine 是纯 .NET 10.0 项目，无框架锁定
  ├── 只有 TrendDB DLL + RocksDB + Redis/ClickHouse 客户端 → 依赖极小
  ├── 独立 Docker 容器，独立端口，独立升级
  └── 未来换 .NET 9 或移植到 Linux arm64，不受 MonitorCenter 约束
```

**唯一的交互通道是接口：**

```
MonitorCenter ──HTTP──► POST /api/ruleengine/sync/full    (全量同步)
                      POST /api/ruleengine/sync/delta    (增量变更)

MonitorCenter ◄──HTTP── GET  /api/ruleengine/alarms/realtime  (查询实时报警)
                        POST /api/ruleengine/alarms/history   (查询历史报警)
```

两边各自维护一套功能相同但相互独立的类定义，通过 JSON 序列化/反序列化交换数据。牺牲了一点代码复用，换来了彻底的部署独立和演进自由。

## 2. 现有 MonitorCenter 架构分析

### 2.1 当前架构

```
SIS.Web Host 进程
  └── MonitorCenter (ABP 插件模块)
        ├── Hangfire 调度层 (DispatchJob 60s, DataSynchronousJob 2s)
        ├── TPL Dataflow 管道
        │   ├── PrerulePipeline (前置规则检查)
        │   ├── MonitorItemPipeline (4 阶段: PreruleCheck → RuleCalculate → EventBuild → CacheSync)
        │   └── DataSynchronousPipeline (批量写入 PostgreSQL)
        ├── 64 分片 ConcurrentDictionary 缓存
        └── SignalR 实时推送 (RealInfoWorker 3s)
```

### 2.2 核心痛点

| 痛点 | 现状 | 影响 |
|------|------|------|
| **无法水平扩展** | 单进程内 TPL Dataflow，并行度 = CPU 核心数 × 8 | 10 万+ 监控项全部挤在一个进程，扩容只能垂直加 CPU |
| **数据链路长** | MonitorSourceManager → gRPC → RealDB RPC → TrendDB | 一个测点值经过 3 跳，延迟叠加 |
| **存储职责混乱** | PostgreSQL 同时承担实时查询和历史存储 | 大表时间范围扫描性能差，实时查询和历史查询互相影响 |
| **配置变更重** | 任一监控项变更触发全量缓存重建 + 管道重启 | 分布式场景不可行 |
| **紧耦合** | 作为 ABP 插件嵌入 SIS.Web Host | 无法独立部署、升级、扩缩容 |

### 2.3 需要保留的资产

RuleEngine 不是推倒重来。以下组件直接复用：

- **9 种规则计算器** (`CalculateRuleExpression`, `CalculateRuleRangeDuration` 等) —— 工业场景特化，久经考验
- **RuleManager 外观模式** —— 根据 RuleType 分发到对应计算器
- **时间归并算法** (`Merge`) —— 同状态时间段合并
- **事件构建逻辑** (`MonitorItemEventBuildBlock`) —— 标准规则、打包值、多状态的统一事件模型
- **极值缓存模式** —— 区间时长规则的状态管理

## 3. 整体架构设计

### 3.1 架构全景图

```
                          ┌──────────────────────┐
                          │    MonitorCenter      │
                          │    (管理平台)          │
                          │  - 监控项 CRUD        │
                          │  - 规则配置           │
                          │  - 状态定义           │
                          └──────────┬───────────┘
                                     │ HTTP API
                                     │ 全量同步 / 增量变更通知
                                     ▼
                        ┌────────────────────────┐
                        │   RuleEngine Master    │
                        │   (Control Plane)      │
                        │                        │
                        │  - 持有全量监视项配置   │
                        │  - Cost-Aware 分区计算  │
                        │  - 配置变更 → 增量推送  │
                        │  - Worker 健康监控      │
                        │  - 对外查询 API        │
                        │  - 多实例 + Redis 选主  │
                        └───────────┬────────────┘
                                    │ gRPC
                    ┌───────────────┼───────────────┐
                    │               │               │
                    ▼               ▼               ▼
          ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
          │  Worker-1    │ │  Worker-2    │ │  Worker-N    │
          │  (Data Plane)│ │  (Data Plane)│ │  (Data Plane)│
          │              │ │              │ │              │
          │ ~10万监控项  │ │ ~10万监控项  │ │ ~10万监控项  │
          │ 本地规则缓存 │ │ 本地规则缓存 │ │ 本地规则缓存 │
          │ RocksDB 状态 │ │ RocksDB 状态 │ │ RocksDB 状态 │
          │ TrendDB 直读 │ │ TrendDB 直读 │ │ TrendDB 直读 │
          └──────┬───────┘ └──────┬───────┘ └──────┬───────┘
                 │                │                │
                 │  写入实时报警  │  写入实时报警  │
                 ▼                ▼                ▼
          ┌──────────┐                  ┌──────────────┐
          │  Redis   │                  │  ClickHouse  │
          │ 实时报警 │                  │  历史报警     │
          └──────────┘                  └──────────────┘
```

### 3.2 Control Plane / Data Plane 职责边界

| 职责 | Control Plane (Master) | Data Plane (Worker) |
|------|----------------------|---------------------|
| 监视项配置存储 | 全量（~2GB，100万项） | 仅自己负责的部分（~200MB） |
| 分区计算 | Cost-Aware 贪心装箱 | 无 |
| 配置变更处理 | 接收 MonitorCenter 通知，计算增量，推送给受影响 Worker | 接收 Master 推送，本地 TryUpdate |
| 故障转移 | 监控 Worker 心跳，触发重分配 | 本地 RocksDB 持久化状态，支持状态迁移 |
| 时序数据获取 | 无 | TrendDB DLL 直读 |
| 规则计算 | 无 | TPL Dataflow 管道，9 种规则计算器 |
| 结果写入 | 无 | Redis (实时) + ClickHouse (历史) |
| 对外查询 | HTTP API (查 Redis/ClickHouse) | 无 |
| 高可用 | 多实例 + Redis 分布式锁选主 | 多 Worker 对等，宕机后 Master 重分配 |

### 3.3 关键设计决策

**决策一：Control Plane 持有全量配置**

理由：
- 分区计算需要全量元数据（itemId, ruleType, refreshInterval）
- Worker 故障转移时，能直接把完整规则配置推给接管 Worker，不需要 Worker 再去 MonitorCenter 拉取
- 解耦 Worker 和 MonitorCenter——Worker 只跟 Master 通信

**决策二：Worker 只存自己的规则，不存全量**

理由：
- 100 万监控项 × 2KB ≈ 2GB，10 个 Worker 各存全量 = 20GB 浪费
- Worker 热路径只需要自己的规则，存全量毫无意义
- 新增监控项时只推送给目标 Worker，不需要广播全量

**决策三：规则配置走本地内存，不走 Redis**

理由：
- 配置读写的比例接近 ∞:1（改一次，读百万次）
- 热路径上每次计算都去 Redis 拿规则 = 每次计算多一次网络往返
- MonitorCenter 已有的 `MonitorItemStore`（64 分片 ConcurrentDictionary）验证了这个模式

## 4. Control Plane（Master）详细设计

### 4.1 核心模块

```
RuleEngine.Master
├── ConfigurationManager      —— 全量配置持有 + 增量更新
├── PartitionManager          —— Cost-Aware 分区计算
├── WorkerManager             —— Worker 注册/心跳/注销
├── FailoverController        —— 故障检测 + 分片接管编排
├── QueryService              —— 对外 HTTP API
└── LeaderElection            —— 多 Master 选主 (Redis)
```

### 4.2 配置管理流程

```
场景 1: 系统启动
===============
MonitorCenter ──HTTP──► Master: POST /sync/full { monitors: [...], version }
                          │
                          ├── 1. ConfigurationManager 全量加载到内存
                          ├── 2. PartitionManager 执行贪心装箱
                          ├── 3. 通过 gRPC 将分区结果推送给各 Worker
                          └── 4. 分区映射表写入 Redis (用于 Master 故障恢复)

场景 2: 监控项变更 (新增/修改/删除)
===================================
MonitorCenter ──HTTP──► Master: POST /sync/delta { added: [...], modified: [...], deleted: [...] }
                          │
                          ├── 1. ConfigurationManager 更新本地全量配置
                          ├── 2. 新增 → 贪心插入当前最轻 Worker (不触发全局重分区)
                          │    修改 → 原地更新，通知对应 Worker
                          │    删除 → 通知对应 Worker 移除
                          └── 3. 定期 (如每天凌晨) 执行一次全局重分区，消除累积的不均衡

场景 3: Worker 宕机
===================
Worker-3 心跳超时 (30s)
  │
  ├── 1. FailoverController 检测到 Worker-3 失联
  ├── 2. 从 Redis 读取 Worker-3 的分片分配表
  ├── 3. 将 Worker-3 的监视项通过有界负载哈希环分散到剩余 Worker
  ├── 4. 从 Worker-3 的 RocksDB 备份 (如有) 恢复状态快照
  └── 5. 推送接管指令 + 规则配置 + 状态快照给接管 Worker
```

### 4.3 Master 高可用

```
                    ┌─────────────┐
                    │    Redis    │
                    │ 分布式锁:    │
                    │ master:leader│
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
              ▼            ▼            ▼
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │Master-1  │ │Master-2  │ │Master-3  │
        │(Leader)  │ │(Standby) │ │(Standby) │
        └──────────┘ └──────────┘ └──────────┘

- Leader 持有分布式锁 (TTL 15s，每 5s 续约)
- Leader 宕机 → 锁过期 → Standby 竞争成为新 Leader
- 分区映射表持久化在 Redis，新 Leader 可直接读取
- 全量配置从 MonitorCenter 重新拉取 (或从本地快照恢复)
```

## 5. Data Plane（Worker）详细设计

### 5.1 Worker 核心循环

```
while (true) {
    ① 从 TrendDB 批量读取本分片所有测点的实时值
    ② 按 RefreshIntervalSecond 分组，判断哪些监控项本周期需要计算
    ③ 对需要计算的监控项:
       a. 从本地 ConcurrentDictionary 读取规则配置
       b. 从本地 RocksDB 读取上一次计算状态 (持续时长、频率计数器等)
       c. 调用 RuleManager.Calculate() 执行规则计算
       d. 若产生新事件:
          - HSET alarm:current:{monitorId} → Redis
          - INSERT INTO alarm_events → ClickHouse
       e. 保存新状态到 RocksDB
    ④ sleep(最小采集间隔) // 或使用更精细的定时器调度
}
```

### 5.2 Worker 内部管道

复用 MonitorCenter 的 TPL Dataflow 模式，但在单个 Worker 内部按规则类型做管道特化：

```
Worker 内部:
  ├── Expression Pipeline:      MaxDOP=2,  BufferSize=10000  (轻量，不需大缓冲)
  ├── RangeDuration Pipeline:   MaxDOP=4,  BufferSize=50000  (需查极值缓存)
  ├── RangeFrequency Pipeline:  MaxDOP=4,  BufferSize=30000  (需遍历频率窗口)
  ├── PackageValue Pipeline:    MaxDOP=4,  BufferSize=20000  (位解析)
  ├── MultiState Pipeline:      MaxDOP=6,  BufferSize=30000  (多状态并发)
  └── IO Pipeline:              MaxDOP=8,  BufferSize=5000   (WallTemperature + InterfaceMonitoring)
                                                                (IO 密集，高并发低缓冲)
```

管道特化与分片策略正交——无论 MonitorItem 通过什么算法分配到本 Worker，进入 Worker 后按规则类型走不同的处理管线。

### 5.3 计算器复用

```csharp
// 直接复用 MonitorCenter 现有代码，无需重写
RuleManager.Calculate(IRealDataRead source, RuleType ruleType,
                      MonitorRuleOptions ruleOptions, DateTime? time)
{
    switch (ruleType)
    {
        case RuleType.Expression:
            return CalculateRuleExpression.Execute(source, ruleOptions);
        case RuleType.RangeDuration:
            return CalculateRuleRangeDuration.Execute(source, ruleOptions, state, time);
        case RuleType.RangeFrequency:
            return CalculateRuleRangeFrequency.Execute(source, ruleOptions, state);
        case RuleType.PackageValue:
            return CalculateRulePackageValue.Execute(source, ruleOptions);
        case RuleType.FeatureValue:
            return CalculateFeatureValue.Execute(source, ruleOptions);
        case RuleType.WallTemperature:
            return CalculateWallTemperature.Execute(source, ruleOptions, time);
        case RuleType.InterfaceMonitoring:
            return CalculateInterfaceMonitoring.Execute(source, tagInfo, time);
        case RuleType.RulePackageValue:
            return CalculateRulePackageValue.Execute(source, ruleOptions);
        case RuleType.RuleMultiStateRangeDuration:
            return CalculateRuleMultiStateRangeDuration.Execute(source, ruleOptions, state);
    }
}
```

## 6. 分片与负载均衡策略

### 6.1 为什么不用纯哈希环

| 特征 | 说明 | 对算法的要求 |
|------|------|-------------|
| **负载异构** | 壁温规则涉及多变量关联计算，表达式规则只是跑一段脚本。单监控项计算成本差异可达 10 倍 | 必须感知负载权重 |
| **状态有成本** | 每个监控项在 Worker 本地 RocksDB 有持续时长、频率计数器。分片迁移 = 状态搬运 | 迁移成本高，计划内操作可接受 |
| **扩缩容是计划内的** | Worker 的新增/下线是运维操作，不是随机上下线 | 不需要哈希环的即时重映射 |
| **采集周期已知** | 每个监控项的 RefreshIntervalSecond 明确定义 | 可以精确预估负载 |

哈希环保证的是** key 数量均匀**，不是**计算负载均匀**。100 个壁温规则恰好哈希到 Worker-2，能把 Worker-2 拖垮。

### 6.2 推荐方案：混合策略

**第一层：Cost-Aware Static Partitioning（稳态，99% 的时间）**

```
成本公式: cost = ComplexityWeight[RuleType] × (1.0 / RefreshIntervalSecond)

ComplexityWeight:
  Expression                       = 1.0   (纯数学运算，< 0.1ms)
  RangeDuration                    = 2.0   (需查极值缓存)
  RangeFrequency                   = 3.0   (需遍历频率窗口)
  PackageValue / FeatureValue      = 2.0   (位解析 / 特征值匹配)
  RulePackageValue                 = 3.0   (多段位打包)
  RuleMultiStateRangeDuration      = 4.0   (多状态并发)
  WallTemperature / InterfaceMonitoring = 5.0   (IO 密集, 额外访问外部服务)

分配算法: 贪心装箱 (Greedy Bin Packing)
  1. 所有监控项按 cost 降序排列
  2. 依次将每个监控项分配给当前总负载最小的 Worker
  3. 结果: 各 Worker 负载差异 < 5%

适用场景:
  - 系统初始启动
  - 运维主动扩容/缩容
  - 定期全局重平衡 (每天凌晨)
```

**第二层：Consistent Hashing with Bounded Loads（瞬态，1% 的时间）**

```
Worker 宕机时:
  1. 宕机 Worker 上的监控项沿哈希环顺时针找下一个可用 Worker
  2. 每个 Worker 设负载上限 capacity = avg_load × (1 + ε)，ε = 0.2
  3. 若第一个候选 Worker 已满，继续沿环找下一个
  4. 1 秒内恢复计算，负载暂时有 10-15% 偏差

适用场景:
  - Worker 意外宕机
  - 不需要 Master 立即做全局重分配 (买时间，从容处理)
```

### 6.3 策略对比

| 维度 | 纯哈希环 | Cost-Aware | **混合策略** |
|------|---------|-----------|-------------|
| 负载均衡度 | 差（不感知成本） | 优（差异 < 5%） | 优 |
| 迁移成本 | 低 | 中（计划内可控） | 计划内中 / 紧急低 |
| 故障恢复速度 | 快 | 慢（需全局重算） | 快 |
| 实现复杂度 | 低 | 中 | 中 |
| 异构负载感知 | 不支持 | 支持 | 支持 |

### 6.4 为什么不能按规则类型分 Worker

这是一个自然的直觉——9 种规则类型，9 个 Worker，各算各的。但不可行：

**问题一：分布天然不均。** Expression 占 ~55%，WallTemperature 占 ~2%。按类型分 Worker 的结果是 Expression Worker 被打爆，6 个 Worker 空转。

**问题二：同类内部无法水平扩展。** 监控项从 10 万涨到 50 万，Expression Worker 永远是单点——除非打破"一个类型一个 Worker"的约定（那方案就没有意义了）。

**问题三：刷新频率才是主导因素。** 1 秒刷新的 Expression 比 60 秒刷新的 WallTemperature 每秒负载高 12 倍。按类型分组完全忽略了这个维度。

**规则类型的正确用法**是作为成本模型的权重输入（`ComplexityWeight` 字典），而不是作为分组键。

### 7.1 数据分层总览

```
┌──────────────┬──────────┬───────────────────┬─────────────────────┐
│  数据类型     │ 存储位置  │ 读写特征           │ 说明                 │
├──────────────┼──────────┼───────────────────┼─────────────────────┤

│              │ 内存+Redis│                   │                      │
└──────────────┴──────────┴───────────────────┴─────────────────────┘
```

### 7.2 Redis 设计

**用途：实时报警快照（分布式共享内存）**

```
Key 设计:
  alarm:current:{monitorId}    → Hash { statusKey, statusName, value, occurTime, configVersion }
  alarm:active                 → Set { monitorId }  当前有活跃报警的监控项集合
  shard:assignment:{shardId}   → String workerId    分片 → Worker 映射
  worker:heartbeat:{workerId}  → String timestamp   Worker 心跳
  master:leader                → String (分布式锁)  Master 选主

TTL 策略:
  实时报警快照: 无 TTL (报警消除时主动删除)
  Worker 心跳: TTL 30s
  Master 锁: TTL 15s (每 5s 续约)
```

**为什么实时报警用 Redis 而不是 Master Fan-out 查询 Worker？**

| 维度 | gRPC Fan-out | Redis 共享存储 |
|------|-------------|---------------|
| 查询延迟 | O(N)，受最慢 Worker 影响 | O(1)，单次 Redis 调用 |
| Worker 宕机 | 该分片数据丢失 | 数据不丢 |
| Worker 重启 | 报警状态全部丢失 | 数据不丢 |
| 多 Master | 需去重 | 天然支持 |

**Redis 在这里的角色不是"缓存"，而是分布式共享内存。** 它的价值不在于快（本地内存更快），而在于它是多 Worker 和多 Master 之间唯一的一块共享空间。

### 7.3 ClickHouse 设计

**用途：历史报警事件存储**

```sql
CREATE TABLE alarm_events (
    monitor_id       String,
    monitor_key      String,
    monitor_name     String,
    status_key       String,
    status_name      String,
    occur_time       DateTime64(3),
    clear_time       Nullable(DateTime64(3)),
    trigger_value    Float64,
    rule_type        UInt8,
    config_version   DateTime64(3),
    worker_id        String,
    shard_id         UInt8,
    date             Date MATERIALIZED toDate(occur_time)
) ENGINE = MergeTree()
PARTITION BY toYYYYMMDD(occur_time)
ORDER BY (monitor_id, occur_time)
TTL occur_time + INTERVAL 365 DAY;

-- 物化视图：按天统计报警量
CREATE MATERIALIZED VIEW alarm_daily_stats
ENGINE = SummingMergeTree()
PARTITION BY toYYYYMM(date)
ORDER BY (date, monitor_id, status_key)
AS SELECT
    date,
    monitor_id,
    status_key,
    count() as alarm_count
FROM alarm_events
GROUP BY date, monitor_id, status_key;
```

### 7.4 RocksDB 设计

**用途：Worker 本地有状态计算的持久化**

```
Key: monitor_id
Value: {
    ruleType: 2,
    lastSatisfiedTime: "2026-07-17T08:00:00Z",
    accumulatedDuration: 45.5,       // 区间时长规则的累计秒数
    frequencyWindow: [t1, t2, t3],   // 区间频率规则的触发时间窗口
    previousStatus: "normal",        // 上一次的计算结果状态
    configVersion: "2026-07-17T00:00:00Z"
}

特性:
  - 每次计算后同步写入 (WriteBatch 批量提交)
  - 宕机恢复时从 RocksDB 读取，不需要重新积累状态
  - 分片迁移时序列化发送给目标 Worker
```

### 7.5 为什么规则配置不放 Redis

| 特征 | 监视项规则配置 | 实时报警状态 |
|------|-------------|------------|
| 读写比 | ∞:1（几乎纯读） | 1:1（每次计算都可能写） |
| 变更频率 | 人工操作，每天可能几次 | 每秒成千上万次 |
| 变更来源 | MonitorCenter 管理界面 | Worker 计算结果 |

把配置放 Redis，意味着每次计算都要走一次网络往返取一个几乎不变的规则。正确的做法是：

```
MonitorCenter 变更 → Master 收到 → 推送给受影响 Worker → Worker 本地内存 TryUpdate
                                                              ↑
                                                        热路径零网络开销
```

## 8. 容错与高可用

### 8.1 故障场景与应对

| 故障场景 | 影响 | 应对策略 | 恢复时间 |
|---------|------|---------|---------|
| Worker 宕机 | 该分片监控项停止计算 | 哈希环自动接管 → Master 异步全局重平衡 | < 1s (接管) / < 5min (重平衡) |
| Master 宕机 | 无法处理配置变更、无法查询 | Redis 选主 → Standby 接管 | < 15s |
| Redis 宕机 | 实时报警查询不可用 | Redis Sentinel / Cluster | < 30s |
| ClickHouse 宕机 | 历史报警写入积压 | 本地 WAL 缓冲，恢复后批量补写 | < 5min |
| TrendDB 不可用 | 所有 Worker 无法获取数据 | 重试 + 告警，不会丢失状态 | 依赖 TrendDB 恢复 |
| 网络分区 | Worker 和 Master 失联 | Worker 继续用本地配置计算，心跳恢复后同步 | 分钟级 |

### 8.2 分片迁移的状态一致性

这是整个系统最有挑战性的部分——区间时长、频率计数器等状态必须在迁移时保持一致。

**计划内迁移（扩容/缩容）：**

```
1. Master 计算出新分区方案
2. 对于需要迁移的监控项 "temp_001" (Worker-2 → Worker-5):
   a. Master 通知 Worker-2: "准备迁移 temp_001，发送最后状态"
   b. Worker-2 停止计算 temp_001，序列化 RocksDB 状态
   c. Worker-2 将状态 + 最新结果发送给 Worker-5
   d. Worker-5 加载状态，初始化 RocksDB
   e. Worker-5 通知 Master: "temp_001 已就绪"
   f. Master 更新分区表，通知 Worker-2 删除 temp_001
   g. Worker-5 开始计算 temp_001
```

**紧急接管（Worker 宕机）：**

```
1. Master 检测到 Worker-2 心跳超时
2. Master 从 Redis 读取 Worker-2 的分片分配表
3. 将 Worker-2 的监控项通过有界负载哈希环分散到剩余 Worker
4. 接管 Worker 从 RocksDB 本地备份 (如有) 或 Worker-2 的共享存储恢复状态
5. 若无备份，从 Redis 中的最后报警快照反推初始状态
6. 接管 Worker 开始计算
```

### 8.3 配置版本号机制

每个监控项的计算结果携带 `configVersion`（MonitorItem 的 `LastModificationTime`）：

- Worker 写入 Redis/ClickHouse 时附带版本号
- 查询方可过滤掉过期版本的数据
- 分片迁移时校验版本一致性

## 10. 部署架构

### 10.1 Worker Native AOT 编译

Worker 作为计算密集型、长时间运行的进程，是 Native AOT 的理想场景：

```
Worker 编译 (Native AOT):
  源码 (.cs) → ILC (IL Compiler) → 原生二进制 (ELF/Mach-O/PE)
  无 JIT、无 IL、无运行时编译

Master 编译 (JIT):
  源码 (.cs) → IL → 运行时 JIT 编译
  保留反射、动态类型加载能力
```

| 指标 | JIT (Master) | Native AOT (Worker) | 收益 |
|------|-------------|---------------------|------|
| 启动时间 | 3-5 秒 | < 0.3 秒 | 故障恢复快 10 倍 |
| 内存占用 (空闲) | ~200MB | ~50MB | 同样硬件跑更多 Worker |
| 容器镜像 | ~200MB (自包含) | ~40MB (裁剪后) | 拉镜像秒级 |
| 首次计算延迟 | JIT 预热 1-3s | 0（无预热） | 接管分片立即算 |
| CPU 吞吐 | 基准 | +10-20% (无 JIT 开销) | 热路径直接受益 |

Docker 部署文件位于项目根目录：

| 文件 | 用途 |
|------|------|
| `docker-compose.yml` | 4 服务编排（Redis + ClickHouse + Master + Worker） |
| `src/Luculent.Sis.RuleEngine.Worker/Dockerfile` | Worker Native AOT 编译 → chiseled 镜像 |
| `src/Luculent.Sis.RuleEngine.Master/Dockerfile` | Master JIT 编译 → aspnet 镜像 |
| `.dockerignore` | 排除 bin/obj/.git/docs/node_modules |
| `deploy/clickhouse/init.sql` | ClickHouse 首次启动建表脚本 |

## 11. 与 Flink 的对比

RuleEngine 本质上是一个"轻量级工业 Flink"，借鉴了其核心思想但不直接使用：

| Flink 概念 | RuleEngine 对应 | 差异 |
|-----------|----------------|------|
| JobManager | Master | 去掉了 Checkpoint 协调，简化为配置管理 + 分区 |
| TaskManager | Worker | 增加了规则类型感知的管道特化 |
| Keyed State | RocksDB (Worker 本地) | 没有分布式 State Backend，状态迁移靠显式序列化 |
| Checkpoint | 分片状态迁移协议 | 计划内迁移，非周期触发 |
| Watermark | 数据时间戳 | 事件时间语义弱化，以处理时间为主 |

不建议直接上 Flink 的原因：
- 现有的 9 种规则计算器是工业场景特化的（壁温计算、打包值解析），用 Flink UDF 重写成本和风险都高
- Flink 部署运维复杂度远高于自研轻量引擎
- 团队对 .NET 技术栈更熟悉

## 13. 核心代码参考

### 13.1 Cost-Aware Partitioner

```csharp
public class CostAwarePartitioner
{
    private static readonly Dictionary<RuleType, double> ComplexityWeight = new()
    {
        [RuleType.Expression]                    = 1.0,
        [RuleType.RangeDuration]                 = 2.0,
        [RuleType.RangeFrequency]                = 3.0,
        [RuleType.PackageValue]                  = 2.0,
        [RuleType.FeatureValue]                  = 2.0,
        [RuleType.WallTemperatureValue]          = 5.0,
        [RuleType.InterfaceMonitoring]           = 5.0,
        [RuleType.RulePackageValue]              = 3.0,
        [RuleType.RuleMultiStateRangeDuration]   = 4.0,
    };

    /// <summary>
    /// 贪心装箱：将监控项分配到 N 个 Worker，最小化最大负载
    /// </summary>
    public Dictionary<string, List<MonitorItem>> Partition(
        List<MonitorItem> items, int workerCount)
    {
        // ① 计算每个监控项的成本并按成本降序排列
        var weightedItems = items
            .Select(item => new
            {
                Item = item,
                Cost = ComplexityWeight[item.RuleType]
                     * (1.0 / Math.Max(item.RefreshIntervalSecond, 1))
            })
            .OrderByDescending(x => x.Cost)
            .ToList();

        // ② 初始化 N 个空的 Worker bucket
        var workers = Enumerable.Range(0, workerCount)
            .ToDictionary(i => i, i => new WorkerBucket(i));

        // ③ 贪心分配：每个监控项给当前总负载最小的 Worker
        foreach (var weighted in weightedItems)
        {
            var lightestWorker = workers.Values
                .OrderBy(w => w.TotalCost)
                .First();

            lightestWorker.Items.Add(weighted.Item);
            lightestWorker.TotalCost += weighted.Cost;
        }

        return workers.ToDictionary(
            kv => $"worker-{kv.Key}",
            kv => kv.Value.Items
        );
    }

    private class WorkerBucket
    {
        public int Id { get; }
        public List<MonitorItem> Items { get; } = new();
        public double TotalCost { get; set; }
        public WorkerBucket(int id) => Id = id;
    }
}
```

### 13.2 Worker 核心计算循环

```csharp
public async Task RunAsync(CancellationToken cancellationToken)
{
    while (!cancellationToken.IsCancellationRequested)
    {
        try
        {
            // ① 从 TrendDB 批量读取本分片所有测点的实时值
            var tagNames = _assignedMonitors.Select(m => m.TagName).ToList();
            var values = await _trendDB.ReadBatchAsync(tagNames);

            // ② 按刷新间隔分组，判断哪些监控项本周期需要计算
            var now = DateTime.UtcNow;
            var dueMonitors = _assignedMonitors
                .Where(m => (now - m.LastCalculateTime).TotalSeconds >= m.RefreshIntervalSecond)
                .ToList();

            foreach (var monitor in dueMonitors)
            {
                if (!values.TryGetValue(monitor.TagName, out var value))
                    continue;

                // ③ 加载上一次计算状态
                var state = await _stateStore.GetAsync(monitor.Id);

                // ④ 执行规则计算 (复用现有逻辑)
                var result = _ruleManager.Calculate(
                    value, monitor.RuleType, monitor.RuleOptions, state, now);

                // ⑤ 若有新事件产生
                if (result.HasNewEvent)
                {
                    await _redis.HSetAsync("alarm:current", monitor.Id, result.ToJson());
                    await _redis.SAddAsync("alarm:active", monitor.Id);
                    await _clickhouse.InsertAsync("alarm_events", result.ToClickHouseRow());
                }
                else if (result.IsCleared)
                {
                    // 报警消除
                    await _redis.HDelAsync("alarm:current", monitor.Id);
                    await _redis.SRemAsync("alarm:active", monitor.Id);
                }

                // ⑥ 保存状态
                await _stateStore.SaveAsync(monitor.Id, result.NewState);
                monitor.LastCalculateTime = now;
            }
        }
        catch (Exception ex)
        {
            _logger.Error(ex, "Worker calculation cycle error");
        }

        await Task.Delay(_minRefreshInterval, cancellationToken);
    }
}
```

## 14. 总结

这套架构的核心思想可以归纳为三句话：

1. **Control Plane 管配置，Data Plane 管计算。** Master 持有全量配置做分区决策，Worker 只存自己的那部分专心计算。谁需要谁存，不多存也不少存。

2. **成本感知分区负责稳态，哈希环应急接管负责瞬态。** 99% 的时间各 Worker 负载差异 < 5%，1% 的故障时间 1 秒内恢复计算。不需要更复杂的算法。

3. **配置走本地内存（变的时候推一把），状态走 Redis（每时每刻都在变），历史走 ClickHouse（列存天生适合）。** 三种数据三种存储，各司其职。

最大的技术挑战不是"能不能实现"，而是有状态计算的故障转移一致性。区间时长、频率计数器这些状态的迁移，需要在 Worker 宕机和计划扩缩容两种场景下都保证正确。这个问题解决好，整套方案就能上生产。
