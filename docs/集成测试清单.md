## Phase 7: 代码改动总结 (2026-07-17)

### 修改文件

| 文件 | 改动 |
|------|------|
| `RuleEngineClient.cs` | 新增 `GetRealtimeAlarmsAsync()` / `QueryHistoryAlarmsAsync()` 方法 + 内部 DTO 类 |
| `RealMonitorAppService.cs` | `GetAllForList` 新增分布式代理分支 `GetAllForListFromRuleEngineAsync()`；用 `monitorKey` 匹配 + DB 直读 |
| `HistoryMonitorAppService.cs` | `GetAllForList` 新增分布式代理分支 `GetAllForListFromRuleEngineAsync()`；用 `monitorKey` 匹配 + DB 直读 |
| `appsettings.json` | RuleEngineUrl 改为 `http://master:11082`（跨 Docker 网络通信） |
| 本文档 | 更新测试结果到 Phase 6 |

### 核心修复

1. **monitorKey vs monitorId 匹配**: Rule Engine 报警的 `monitorId` 是内部 ID，与 Monitor Center 的 DB ID 不同。改用 `monitorKey` ↔ `Key` 匹配
2. **分布式模式下 MonitorItemStore 缓存为空**: 本地计算引擎不启动导致缓存不填充。改用 `IRepository.GetAll()` 直接从 DB 读取
3. **跨 Docker 网络通信**: `sis-service` 与 `ruleengine-master` 在不同网络，需 `docker network connect`
4. **DLL 部署**: Plugins 目录无 volume mount，必须 `docker cp` 进容器
5. **ClickHouse alarm_events 表**: 历史查询需要此表存在，当前未创建
| Phase 6: 边界与异常 | 1/4 | on-changed 降级回退已验证

## 测试中发现的问题与修复

1. **MonitorCenterClient DI 双重注册**: `AddHttpClient<T>` 已注册为 typed client，额外的 `AddSingleton<T>` 导致 BaseAddress 为 null。修复：移除 AddSingleton。
2. **ABP 插件 API URL 前缀**: 插件模块的 API 使用模块名 (如 `monitorcenter`) 作为 URL 前缀。URL 正则从 DLL 名称提取。
3. **API 方法类型**: ABP 动态 Web API 对无参数的 `Get*` 方法暴露为 HTTP GET，而非 POST。
4. **GetHisEvent 未实现分布式代理**: Monitor Center 中只有 `GetRealEvent` 实现了到规则引擎的代理。
5. **端口冲突 (11081)**: 本地 Luculent 进程占用 11081，导致外部 IP 请求被路由到错误进程。修复：切换 Rule Engine 到 11082。
6. **JSON 反序列化 camelCase 映射**: `MonitorCenterClient` 使用默认 `JsonSerializer.Deserialize` 不应用 camelCase，导致 `id`→`Id` 映射失败。修复：添加 `PropertyNameCaseInsensitive = true`。
7. **nullable 字段**: `ManualFlag` 和 `FailureCount` 在 Monitor Center 返回 `null`，但 `MonitorConfig` 声明为 `int`。修复：改为 `int?`。
