## 一、方案成熟度评价

**方向完全正确。** 这个方案本质上是在做一个"轻量级 Flink"——事件驱动的分布式规则计算引擎。它是 MonitorCenter 现有架构演进的自然方向，不是不成熟的想法。

## 二、现有 MonitorCenter 架构的痛点

### 2.1 单体进程内管道，无法水平扩展

```
当前：SIS.Web Host 进程
  └── MonitorCenter (ABP 插件)
        ├── TPL Dataflow Pipeline (单机 4-32 并行度)
        ├── 所有 10 万个监控项挤在一个进程里算
        └── 扩容只能加 CPU 核心，不能加节点
```

### 2.2 数据获取链路太长

```
当前链路：
  MonitorSourceManager → gRPC → RealDB RPC Service → TrendDB
  3 跳才拿到一个测点值
```

### 2.3 计算结果存储不合理

```
当前：
  内存缓存 (MonitorDataStore 64分片)
  → DataSynchronousPipeline (每2秒批量写入)
  → PostgreSQL (ssmcmonitordat / ssmceventdat / ssmccureventdat)
```
PostgreSQL 承载了实时查询和 historical 存储双重职责，大表时间范围查询性能瓶颈明显。

### 2.4 配置变更触发全量重建

```
MonitorItemChangeEventHandler → 1秒 debounce → RuntimeManager.Initialize()
→ 全量重建 MonitorItemStore 缓存 + 重启 3 条管道
```
任何一条监控项变更都会触发全量重建，分布式场景不可行。

## 四、关键风险与应对策略

### 4.1 哈希环 + 有状态计算（最大风险）

现有 9 种规则中大部分是有状态的：

```
区间时长规则 (RangeDuration):
  示例: "温度 > 100 且持续 60 秒 → 报警"
  状态: 上次满足条件的时间戳 + 已累计的持续时长

区间频率规则 (RangeFrequency):
  示例: "1 小时内压力 > 10 的次数 >= 5 → 报警"
  状态: 滑动窗口内的触发次数历史

壁温规则 (WallTemperature):
  多变量关联计算，有历史数据依赖
```

**Flink 的做法**是把状态存在 RocksDB（本地）或远程 State Backend，通过 Checkpoint 做容错。

**Worker 宕机后哈希环 Rebalance，状态怎么保证不丢？**

应对策略：

- Worker 本地嵌入 RocksDB 或 SQLite 做轻量状态存储
- Master 监控 Worker 心跳，Worker 宕机后触发分片迁移
- 分片迁移时源 Worker 先把状态序列化发送给目标 Worker，再切换分片归属
- 每个计算结果带上 `configVersion`（MonitorItem 的 `LastModificationTime`），避免旧版本数据覆盖新版本

### 4.2 配置同步的时序问题

**场景 1：监控项 A 被删除**
```
T0: Master 通知 Worker-3 "删除 A"
T1: Master 等待 Worker-3 确认
T2: 如果 Worker-3 在 T0~T2 之间刚好算出了 A 的最新事件？
→ 需要"栅栏"机制：Worker 确认删除后，Master 不再接受该监控项的任何新数据写入
```

**场景 2：监控项 A 的规则变更**
```
T0: MonitorCenter 修改了 A 的规则类型（表达式 → 区间时长）
T1: Master 通知 Worker-3
T2: Worker-3 还在用旧规则计算最后一批数据
→ 新旧规则计算结果可能同时存在
```

**场景 3：哈希环扩容**
```
T0: 4 个 Worker → 6 个 Worker
T1: 监控项 "temp_001" 从 Worker-2 迁移到 Worker-5
T2: Worker-2 上 "temp_001" 的"持续时长计数器"历史状态需要迁移
```

应对策略：

- 每个监控项附带 `configVersion`（复用 `LastModificationTime`）
- Worker 写入 Redis/ClickHouse 时带上版本号
- 哈希环迁移时，源 Worker 先序列化状态发送给目标 Worker，再切换
- Master 维护分片分配版本号（`shardEpoch`），防止脑裂

### 4.3 Master 单点问题

Master 负责分片管理，如果 Master 宕机，整个系统不可用。

应对策略：

- Master 本身做无状态设计，所有关键状态（分片分配表、Worker 列表）存 Redis
- 部署多个 Master 实例，通过 Redis 分布式锁选举 Leader
- Worker 本地缓存分片分配信息，Master 短暂不可用时 Worker 继续按本地缓存工作

## 五、分片算法的选择：哈希环够用吗？

### 5.1 先看这个场景的特殊性

哈希环是一个通用方案，但你这个工业监控场景有几个特殊性，决定了算法选择不能只看"均衡性"：

| 特征 | 说明 | 对算法的要求 |
|------|------|-------------|
| **负载不均匀** | 壁温规则涉及多变量关联计算，表达式规则只是跑一段脚本。单个监控项的计算成本差异可达 10 倍以上。 | 必须感知负载权重，纯随机哈希会导致严重的木桶效应 |
| **状态有成本** | 每个监控项在 Worker 本地 RocksDB 里有持续时长、频率计数器等状态。分片迁移 = 状态搬运。 | 迁移监控项的成本远高于普通 KV，要尽可能减少迁移 |
| **扩缩容是计划内的** | 不像 CDN 节点随时上下线，Worker 的新增/下线是运维操作，可以接受"稍慢但保证正确"的迁移 | 不需要哈希环那种"K/N 个 key 快速重映射"的即时性 |
| **采集周期已知** | 每个监控项的 `RefreshIntervalSecond` 是明确定义的，10 万个 × 60s 和 1000 个 × 1s 的负载完全不同 | 可以做精确的负载预估，不需要运行时自适应的动态均衡 |

### 5.2 候选算法对比

#### 方案 A：一致性哈希环（原始方案）

```
监控项 "temp_001" → hash("temp_001") → 定位到环上 → 顺时针找到最近的 Worker
```

| 优点 | 缺点（本场景下） |
|------|-----------------|
| 新增 Worker 时只迁移 K/N 个 key | 复杂监控项可能随机扎堆在一个 Worker 上，无人察觉 |
| 实现简单，有现成库 | "一致性"的价值被高估了——节点变化是计划内的，不是随机的 |
| 虚拟节点缓解不均 | 虚拟节点只均匀了 key 数量，不是计算负载 |

**关键问题**：哈希环保证的是"key 数量均匀分布"，不是"计算负载均匀分布"。如果有 100 个壁温规则恰好哈希到 Worker-2，而 Worker-1 上全是简单的表达式规则，Worker-2 会被拖垮。

#### 方案 B：Rendezvous Hashing（最高随机权重哈希）

```
对于每个监控项 m，计算它在每个 Worker 上的权重 w = hash(m, workerId)
选择权重最高的 Worker 来处理 m
```

| 优点 | 缺点 |
|------|------|
| 天然比一致性哈希更均匀 | 每次查询需要 O(N) 计算（N = Worker 数量），但 N 很小（< 20） |
| 不需要虚拟节点 | 新增 Worker 时，迁移量 = M/(N+1)，略高于一致性哈希的最优情况 |
| 添加 Worker 时，只影响"新 Worker 权重最高"的监控项 | 不支持带权重的哈希 |

**评价**：比一致性哈希更好，但同样不解决负载异构问题。

#### 方案 C：Cost-Aware Static Partitioning（成本感知静态分片）

```
核心思想：不再哈希 key 来决定归属，而是 Master 主动分配。

1. 计算每个监控项的"计算成本"
   cost = RuleComplexityWeight[RuleType] × (1.0 / RefreshIntervalSecond)

   其中 RuleComplexityWeight:
     Expression              = 1pt
     RangeDuration           = 2pt
     RangeFrequency          = 3pt
     PackageValue            = 2pt
     FeatureValue            = 2pt
     WallTemperature         = 5pt  (多变量关联)
     InterfaceMonitoring     = 2pt
     RulePackageValue        = 3pt
     RuleMultiStateRangeDuration = 4pt

2. 将所有监控项按 cost 降序排列

3. 使用贪心装箱（Greedy Bin Packing）将监控项分配给 Worker：
   遍历排序后的监控项列表，每次选择当前总负载最小的 Worker

4. 分配结果序列化存储到 Redis：
   shard:assignment:{monitorItemId} → workerId
```

| 优点 | 缺点 |
|------|------|
| 各 Worker 的计算负载精确均衡 | 新增 Worker 时需要重新装箱（全局重算） |
| 不会出现"热 Worker"木桶效应 | 迁移量可能比哈希环大 |
| 装箱后各 Worker 的负载差异 < 5% | 需要预定义各规则类型的复杂度权重 |

#### 方案 D：Consistent Hashing with Bounded Loads（Google 的带负载上限哈希）

```
Google 论文 "Consistent Hashing with Bounded Loads" (2017):

1. 每个 Worker 设置一个负载上限 capacity = average_load × (1 + ε)
   其中 ε 是可容忍的不均衡系数（如 0.1 表示允许 10% 偏差）

2. 常规哈希：monitorItem → 哈希环定位到 Worker-A

3. 如果 Worker-A 已满（负载 > capacity），
   沿着环顺时针找下一个有空余的 Worker

4. 当 Worker-A 的负载回落到 capacity 以下时，恢复接收
```

| 优点 | 缺点 |
|------|------|
| 只需重映射超载 Worker 上的 key | 实现比普通哈希环复杂 |
| 节点故障时自动重新分配 | 仍然需要定义"负载"的度量方式 |
| 迁移量最小 | 长尾监控项仍然会被挤到不理想的 Worker |

### 5.3 推荐方案：混合策略

对于你的场景，最佳选择不是单选，而是**分层组合**：

```
第一层：Cost-Aware Static Partitioning（计划内使用）
  → 初始分配、Worker 扩容、Worker 缩容等计划内操作
  → Master 运行贪心装箱算法，一次性算出全局最优分配
  → 然后通过协调迁移协议（先双写、再切换）平滑完成

第二层：Consistent Hashing with Bounded Loads（应急使用）
  → Worker 宕机时的紧急 failover
  → 宕机 Worker 上的监控项沿着哈希环自动分散到其他 Worker
  → 不需要 Master 立即做全局重分配（买时间来从容处理）
```

**为什么这样设计？**

```
场景 1：运维人员加了 3 台 Worker（计划内）
  → 走第一层：Master 全局重装箱，各 Worker 负载差异 < 5%
  → 迁移量大但可控，因为是有计划的低峰操作

场景 2：Worker-3 半夜宕机了（紧急）
  → 走第二层：哈希环自动 failover，1 秒内恢复计算
  → 负载可能暂时不均衡（10-15% 偏差），但不会丢数据
  → 运维第二天上班再从容做全局重分配
```

### 5.4 对比总结

| 维度 | 纯哈希环 | Rendezvous | Cost-Aware | **混合策略（推荐）** |
|------|---------|-----------|------------|---------------------|
| 负载均衡度 | ⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 迁移成本 | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐ (计划内) / ⭐⭐⭐⭐ (紧急) |
| 实现复杂度 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐ |
| 故障恢复速度 | ⭐⭐⭐ | ⭐⭐⭐ | ⭐ | ⭐⭐⭐⭐ |
| 异构负载感知 | ❌ | ❌ | ✅ | ✅ |
| 适合本场景 | 及格 | 良好 | 优秀（仅计划内） | **最优** |

### 5.5 核心代码参考：Cost-Aware Partitioning

```csharp
public class CostAwarePartitioner
{
    // 规则复杂度权重
    private static readonly Dictionary<RuleType, double> ComplexityWeight = new()
    {
        [RuleType.Expression]              = 1.0,
        [RuleType.RangeDuration]           = 2.0,
        [RuleType.RangeFrequency]          = 3.0,
        [RuleType.PackageValue]            = 2.0,
        [RuleType.FeatureValue]            = 2.0, 
        [RuleType.WallTemperature]         = 5.0,
        [RuleType.InterfaceMonitoring]     = 2.0,
        [RuleType.RulePackageValue]        = 3.0,
        [RuleType.RuleMultiStateRangeDuration] = 4.0,
    };

    /// <summary>
    /// 贪心装箱：将监控项分配到 N 个 Worker，最小化最大负载
    /// </summary>
    public Dictionary<string, List<MonitorItem>> Partition(
        List<MonitorItem> items, int workerCount)
    {
        // ① 计算每个监控项的成本并按成本降序排列
        var weightedItems = items
            .Select(item => new
            {
                Item = item,
                Cost = ComplexityWeight[item.RuleType] 
                     * (1.0 / Math.Max(item.RefreshIntervalSecond, 1))
            })
            .OrderByDescending(x => x.Cost)
            .ToList();

        // ② 初始化 N 个空的 Worker bucket
        var workers = Enumerable.Range(0, workerCount)
            .ToDictionary(i => i, i => new WorkerBucket(i));

        // ③ 贪心分配：每个监控项给当前总负载最小的 Worker
        foreach (var weighted in weightedItems)
        {
            var lightestWorker = workers.Values
                .OrderBy(w => w.TotalCost)
                .First();
            
            lightestWorker.Items.Add(weighted.Item);
            lightestWorker.TotalCost += weighted.Cost;
        }

        // ④ 返回分配结果
        return workers.ToDictionary(
            kv => $"worker-{kv.Key}",
            kv => kv.Value.Items
        );
    }
}
```

### 5.6 能否单纯按规则类型分类计算？

**直觉**：9 种规则类型，9 个 Worker，每个 Worker 只算一种规则。不需要哈希环、不需要一致性哈希、不需要成本感知分区——简单直接。

**结论：不行。** 但规则类型在成本模型中是关键输入。下面逐层分析。

#### 5.6.1 问题一：规则类型分布极度不均

从 MonitorCenter 实际代码和工业场景经验来看，规则类型的分布大致是：

```
Expression (表达式):           ████████████████████  ~55%
RangeDuration (区间时长):      ████████              ~20%
RangeFrequency (区间频率):     ████                  ~10%
FeatureValue (特征值):         ██                    ~5%
PackageValue (打包值):         ██                    ~4%
RuleMultiStateRangeDuration:   █                     ~3%
WallTemperature (壁温):        █                     ~2%
InterfaceMonitoring (接口):    █                     ~1%
RulePackageValue (多打包值):   <1%
```

如果按规则类型各分一个 Worker：

```
Worker-Expression:   55000 项 → 直接打满，延迟飙升
Worker-RangeDuration: 20000 项 → 繁忙
Worker-RangeFrequency: 10000 项 → 适中
Worker-FeatureValue:    5000 项 → 半空闲
Worker-WallTemperature:  2000 项 → 基本空转
Worker-InterfaceMonitoring: 1000 项 → 完全空转
```

**6 个 Worker 闲着，1 个 Worker 被打爆，系统整体延迟由最慢的 Worker 决定。** 这比不分片更糟糕——分片的目标是均摊负载，按规则类型分片反而制造了更严重的倾斜。

#### 5.6.2 问题二：同类规则内部无法水平扩展

假设未来监控项从 10 万增长到 50 万，Expression 规则从 5.5 万增长到 27 万。按规则类型分片，你只能给 Expression Worker 加更多 CPU 核心（垂直扩展），但**不能增加第二个 Expression Worker**，因为类型和 Worker 是 1:1 绑定的。

你要么打破"一个类型一个 Worker"的约定（那这个方案的意义就没了），要么接受 Expression Worker 永远单点。

#### 5.6.3 问题三：同类型内部的成本差异依然巨大

同样都是 Expression 规则：

```csharp
// 轻量 Expression：单次计算 < 0.1ms
"x > 100"

// 重量 Expression：单次计算 5-10ms
"Math.Sqrt(Math.Pow(x, 2) + Math.Pow(y, 2)) * Math.Log(z) + CustomFunction(a, b, c)"
```

同样都是 RangeDuration，刷新间隔从 1 秒到 300 秒不等，实际计算频率差 300 倍。

**规则类型只是一个维度，不能单独决定负载。**

#### 5.6.4 问题四：忽略刷新频率这个主导因素

一个每 1 秒刷新一次的 Expression 规则，实际计算成本 = `1.0 * (1.0/1) = 1.0`。

一个每 60 秒刷新一次的 WallTemperature 规则，实际计算成本 = `5.0 * (1.0/60) = 0.083`。

**在每秒的计算负载中，轻量 Expression 比重量 WallTemperature 高 12 倍。** 按类型分组完全忽略了频率这个主导维度。

#### 5.6.5 规则类型的真正价值：调度提示而非分组键

虽然不能作为分组键，但规则类型在以下三个场景中非常有用：

**场景一：成本模型的关键输入**

`CostAwarePartitioner.ComplexityWeight` 字典已经利用了规则类型：

```csharp
ComplexityWeight[RuleType.Expression]              = 1.0  // 纯数学运算
ComplexityWeight[RuleType.RangeDuration]           = 2.0  // 需要查极值缓存
ComplexityWeight[RuleType.RangeFrequency]          = 3.0  // 需要遍历频率窗口
ComplexityWeight[RuleType.RuleMultiStateRangeDuration] = 4.0  // 多状态并发
ComplexityWeight[RuleType.WallTemperature]         = 5.0  // 需要额外查 RealDB
```

这保证了 WallTemperature 规则的 2000 个监控项和 Expression 规则的 2000 个监控项**不会被视为同等负载**。

**场景二：Worker 内部的管道特化**

同一个 Worker 内部，不同规则类型可以用不同的处理管线：

```
Worker-1 内部:
  ├── Expression Pipeline:      MaxDOP=2, BufferSize=10000  (轻量，不需要大缓冲)
  ├── RangeDuration Pipeline:   MaxDOP=4, BufferSize=50000  (需要查极值缓存)
  └── WallTemperature Pipeline: MaxDOP=8, BufferSize=5000   (IO密集，高并发)
```

这与分片策略正交——无论用一致性哈希还是成本感知分区，管道特化都可以叠加。

**场景三：资源隔离（特殊规则类型）**

对于调用外部服务的规则类型（如 InterfaceMonitoring 依赖 HTTP 健康检查），可以做**部分类型隔离**：

```
Worker-1,2,3,4,5:  算 Expression, RangeDuration, RangeFrequency, FeatureValue, 
                    PackageValue, RulePackageValue, RuleMultiStateRangeDuration
                    (纯计算型 Worker，无外部依赖)
                    
Worker-6,7:         算 WallTemperature, InterfaceMonitoring
                    (IO 型 Worker，可能需要访问外部接口)
```

这不是按类型分组做负载均衡，而是**按 IO 特征做资源隔离**——避免一个慢 IO 调用阻塞所有纯计算任务。

#### 5.6.6 正确的策略组合

| 策略 | 作用 | 层级 |
|------|------|------|
| 成本感知静态分区 | 将监控项按 `成本 = 复杂度权重 × (1/刷新间隔)` 均匀分配到 Worker | 跨 Worker 分配 |
| 一致性哈希 + 有界负载 | 紧急故障转移时最小化重分配 | 跨 Worker 容错 |
| 规则类型感知的管道特化 | Worker 内不同规则用不同并行度和缓冲策略 | Worker 内部优化 |
| IO/CPU 规则分离 | 隔离有外部依赖的规则类型到专用 Worker | 资源隔离 |

#### 5.6.7 一句话总结

**"按规则类型分 Worker"的直觉方向是对的——你意识到了不同规则类型的计算成本不同。但正确做法是把规则类型编码为成本权重，而不是用作分组键。分组键必须保证负载均匀，成本权重正是为了让这个均匀成为可能。**

### 5.3 Worker 核心循环

```csharp
// 对标现有 DataSourceCollectionJob + MonitorItemPipeline 的逻辑
public async Task RunAsync(CancellationToken cancellationToken)
{
    // ① 从 TrendDB 拉取分片内所有监控项的实时值
    var tagNames = _assignedMonitors.Select(m => m.TagName).ToList();
    var values = await _trendDB.ReadBatchAsync(tagNames);

    foreach (var monitor in _assignedMonitors)
    {
        // ② 获取测点值
        var value = values[monitor.TagName];

        // ③ 加载上一次计算状态（持续时长、频率计数器等）
        var state = await _stateStore.GetAsync(monitor.Id);

        // ④ 复用现有 CalculateRule* 计算逻辑
        var result = _ruleManager.Calculate(
            value, monitor.RuleType, monitor.RuleOptions, state, DateTime.UtcNow);

        // ⑤ 如果有新事件产生
        if (result.HasNewEvent)
        {
            await _redis.HSetAsync("alarm:current", monitor.Id, result.ToJson());
            await _redis.SAddAsync("alarm:active:set", monitor.Id);
            await _clickhouse.InsertAsync("alarm_events", result.ToClickHouseRow());
        }

        // ⑥ 保存状态
        await _stateStore.SaveAsync(monitor.Id, result.NewState);
    }
}
```

## 七、Redis 的必要性分析：数据放内存还是 Redis？

### 6.1 先区分三类数据

在讨论要不要 Redis 之前，先把"数据"按性质分开：

| 数据类型 | 生产者 | 消费者 | 生命周期 | 典型例子 |
|---------|--------|--------|----------|---------|
| **Worker 计算状态** | Worker 自己 | Worker 自己 | 跨计算周期 | 持续时长计数器、频率滑动窗口、上次满足条件的时间戳 |
| **当前报警快照** | Worker | Master（查询API） | 从产生到消失 | "temp_001 当前处于超温报警状态，触发值 105.3" |
| **报警变更事件** | Worker | Master（推送到外部系统） | 一次性 | "temp_001 从正常进入超温报警" |

### 6.2 Worker 计算状态 → 必须在本地内存

这一类数据没有争议。每个计算周期 Worker 都要读写这些状态（读上次的累计时长、写本次的新累计时长），访问频率等于数据采集频率。

```
示例：10万个监控项 × 60秒采集周期 = 每秒 1667 次状态读写
```

如果走 Redis，每次计算都要一次网络往返（~0.5ms），这在大规模场景下会成为瓶颈。所以 **Worker 计算状态必须用本地内存 + RocksDB 持久化**。

结论：**不需要 Redis。**

### 6.3 当前报警快照 → 这才是 Redis 讨论的核心

### 方案 A：不用 Redis，Worker 内存 + Master gRPC Fan-out

```
查询请求: "给我所有实时报警"

Master ──gRPC──► Worker-1 ──返回 235 条
       ──gRPC──► Worker-2 ──返回 198 条
       ──gRPC──► Worker-3 ──返回 312 条

Master 在内存中合并 3 个结果，返回给调用方
```

**优点：**
- 不引入 Redis，减少一个基础设施依赖和运维成本
- 数据源头唯一（Worker），没有"Redis 和 Worker 内存不一致"的问题

**缺点：**
- 每次查询要 Fan-out 到所有 Worker，Worker 数量越多延迟越高（P99 受最慢 Worker 影响）
- Worker 宕机时，该分片的报警数据直接丢失（查询返回不完整结果）
- Worker 重启后，内存中所有报警快照丢失——系统无法知道"重启前有哪些报警还在活跃"
- 报警列表是高频查询（前端每 3 秒轮询一次），每个查询都要惊扰所有 Worker

### 方案 B：用 Redis 作为当前报警的共享存储

```
Worker 计算完毕:
  redis.HSET("alarm:current", monitorId, snapshot)   // 更新快照
  redis.SADD("alarm:active", monitorId)              // 加入活跃集合

Master 查询:
  ids = redis.SMEMBERS("alarm:active")               // 拿到所有活跃报警ID
  snapshots = redis.HMGET("alarm:current", ids)      // 批量拿快照
```

**优点：**
- Master 查询时完全不打扰 Worker，一个 Redis 调用搞定
- Worker 宕机/重启，报警快照不丢
- 天然支持多 Master 实例（都读同一个 Redis）

**缺点：**
- 多一个 Redis 要维护（但你现有架构已经有 Redis 了——Hangfire 在用）
- Worker 写 Redis 失败时需要重试逻辑（但 Redis 本身足够稳定）

### 6.4 对比结论

| 维度 | 方案 A (无 Redis, gRPC Fan-out) | 方案 B (Redis 共享存储) |
|------|-------------------------------|------------------------|
| 查询延迟 | O(N) Worker 数量，P99 不可控 | O(1)，单次 Redis 调用 |
| Worker 宕机 | 该分片报警数据丢失 | 数据不丢 |
| Worker 重启 | 报警状态全部丢失 | 数据不丢 |
| 多 Master | 需做结果去重 | 天然支持 |
| 引入新依赖 | 无 | 无（现有架构已有 Redis） |
| 数据一致性 | 强一致（查源头） | 最终一致（ms 级延迟） |
| 适合规模 | Worker ≤ 3 台 | 任意规模 |

### 6.5 最终建议：Redis 要用，但只用它存"当前快照"

```
┌─────────────────────────────────────────────┐
│              数据分层存储策略                  │
├──────────────┬──────────┬───────────────────┤
│ Worker 计算状态 │ 本地内存   │ RocksDB 异步持久化 │
│ 当前报警快照    │ Redis    │ Hash + Set       │
│ 报警变更事件    │ 直接投递  │ gRPC 推送给 Master │
│ 历史报警       │ ClickHouse │ 永久存储          │
└──────────────┴──────────┴───────────────────┘
```

**Redis 在这里的角色不是"缓存"，而是分布式共享内存。** 它的价值不在于快（本地内存更快），而在于它是多个 Worker 和 Master 之间唯一的一块共享空间。

**如果在阶段 1 你只有 1 个 Worker，Redis 确实可以省略。** Worker 自己内存里存报警快照，Master 连上这个 Worker 查询即可。但一旦进入阶段 2（多 Worker），Redis 就会从"可省略"变成"必须品"。

## 八、关于 Flink 的参考建议

Flink 的核心思想值得借鉴，但不建议直接上 Flink：

| Flink 概念 | 对应本方案 | 借鉴价值 |
|-----------|-----------|----------|
| JobManager | Master | 任务调度、Checkpoint 协调 |
| TaskManager | Worker | 数据计算执行 |
| Keyed State | RocksDB 本地状态 | 有状态计算的状态存储 |
| Checkpoint | 分片状态序列化 | 故障恢复 |
| Watermark | 数据时间戳 | 乱序数据处理 |

不建议直接上 Flink 的原因：

- 现有的 9 种规则计算器 `CalculateRule*` 是工业场景特化的（壁温计算、打包值解析、接口监控），用 Flink UDF 重写成本高
- Flink 部署运维复杂度远高于自研轻量引擎
- 团队对 .NET 技术栈更熟悉，Flink 是 Java/Scala 生态

## 十、总结

**这个方案不是不成熟，而是工业级分布式规则引擎的标准做法。** 描述的思路本质上就是 "Control Plane + Data Plane + Cost-Aware 分片 + Redis/ClickHouse 存储"，和 Flink 的 JobManager/TaskManager 架构一脉相承，只是针对工业监控场景做了特化。

最大的技术挑战不是"能不能实现"，而是：

1. **有状态计算在 Worker 故障转移时的状态一致性**
2. **分片迁移时的平滑过渡（计划内贪心装箱 + 应急哈希环 failover 双模切换）**
3. **Master 自身的高可用**

这三个问题解决好，整套方案就能上生产。
